# rezidnt — subject taxonomy

**Taxonomy version:** v0 · **This file is the canonical copy.** `docs/rezidnt-architecture.md` Appendix B is the excerpt; when they diverge, this file wins and the appendix is stale.
**Provenance:** bootstrapped 2026-07-16 from architecture doc v0.2 — Appendix B, plus the DR-001 amendments (native harness telemetry subjects; sole-allocator worktree model).
**Custody:** this file is edited only through `/subject` inside a warden-gated ontology session; direct edits are blocked by the ontology-gate hook. Every change appends one line to the changelog at the bottom. Any change that amends a BINDING item routes through `/dr` first.

## Grammar and change discipline (BINDING)

- Subjects are dot-namespaced: `noun.verb[.qualifier]`.
- Past tense for facts (`worktree.allocated`, `gate.passed`); present tense for state deltas (`agent.status.changed`).
- Subjects are **never renamed** — deprecation only. A deprecated subject's name stays reserved forever.
- Payloads evolve **additively**. A breaking payload change mints `v+1` for that subject, and every live reducer must handle all live versions.
- Payloads ride the Event envelope (doc §5): `v: u16` is the per-subject payload schema version; payloads are JSON, hard-capped at 32 KiB (I2). Larger content becomes a CAS ref (`CasRef { hash, bytes, mime }`), never inline bytes.
- Taxonomy v0 starts every subject at `v = 1`.

**Grammar note (grandfathered):** six v0 subjects keep their Appendix B forms verbatim even though they do not strictly parse as `noun.verb`: `worktree.conflict`, `gate.inconclusive`, `diff.ready`, `daemon.warning`, `daemon.error`, `agent.message`. They are canonical and will never be renamed; strict `noun.verb[.qualifier]` applies to all new subjects from v0 forward.

### badge

**Operator badge (S3 note):** doc §12 badges are per-`AgentRun` capability tokens; S3 additionally mints one daemon-lifetime **operator badge** for local human clients, announced via the 0600 MCP lockfile (`{pid, port, url, badge}`) — possession of the file is possession of the capability. This is a §12/DEFAULT security-layer reading, pinned by the S3 board; the lockfile shape itself is discovery metadata, **not fabric surface**, and is documented at doc level (scribe scope). The concept is blessed here only insofar as `badge.*` semantics must eventually accommodate a badge scoped to the daemon lifetime rather than a run. DR-005 (owner-ratified 2026-07-17) settled the coupled badge questions: the operator badge is blessed as a distinct daemon-lifetime badge class (DEFAULT), and the `badge.issued` emit-or-drop question is **decided — drop the emitter** (attribution rides `agent.spawned.badge_id`; see the `badge.issued`/`badge.revoked` rows below).

**§12 badge rule (out of ontology scope — recorded for cross-reference only):** DR-005 also narrows the §12 rule to "a badge is required on every **state-mutating** call" (spawn, open, merge), with interrogation (`gate_explain`, `gate why`) and `tail_events` treated as **read-class/unbadged** so a blocked agent can always read *why* it was blocked (I6). That rule lives in doc §12, not in this taxonomy — the ontology carries subjects and payload shapes, not the call-badging policy — so no ontology entry is minted for it; it is noted here purely so the badge section and DR-005 do not read as divergent.

| Subject | v | Emitter | Semantics |
|---|---|---|---|
| `badge.issued` | 1 | **none in v1 (DR-005)** | Reserved subject: a per-run capability badge is minted (256-bit token scoped to `{workspace, verb set, expiry}`, doc §12, injected at spawn per DR-001) — but this subject has **NO emitter in v1**. Issuance is attributable via `agent.spawned.badge_id` (DR-005; `bins/rezidentd/src/runs.rs`), so no separate `badge.issued` fact is emitted. Subject stays reserved (never renamed); the emitter is DEFAULT-droppable and cheap to revisit if a run-independent delegation use case appears. |
| `badge.revoked` | 1 | **none in v1 (DR-005)** | Reserved subject, consistent with `badge.issued`: **NO emitter in v1** (no revocation path exists as built). Revocation would mean mutating calls bearing the badge are refused thereafter. Reserved (never renamed); an emitter arrives only if/when revocation is built. |

### integrity

DR-006 (owner-ratified 2026-07-17) chose a **dedicated subject** for replay divergence over reusing the broad `daemon.error` bucket AND over polluting the gate vocabulary — the integrity-of-log signal (recorded ≠ replayed) must be precisely queryable, and integrity-of-log ≠ gate verdict. This new top-level noun is the ratified intent, not a folding under `gate.*`.

| Subject | v | Emitter | Semantics |
|---|---|---|---|
| `integrity.alarm` | 1 | daemon (single writer; DR-006 routes the `debrief` append through the daemon's sole log writer, I3) | A recorded verdict on the log diverged from the verdict re-derived on replay (`debrief` replay over log + CAS). Recorded ≠ replayed means verifier nondeterminism (a verifier bug) or an altered log (§12); raised, named, never silently reconciled. Lands a **durable fact** on the log (DR-006) — strictly additive to the CLI `debrief --json` report (`report.alarms[]`, exit 3) which stays correct. Folded by `rezidnt-state` into `agent_runs[run].integrity_alarms`, so it is **not** a dead-letter subject. |

### permit

The **pre-hoc authorization axis** (DR-008 permit-engine pivot; scope extended by DR-009). rezidnt is the in-daemon Policy Decision Point (PDP); the harness `PreToolUse` hook is the Policy Enforcement Point (PEP). A `permit`-verifier is a verifier on the new `permit` gate lifecycle point — **the gate engine *is* the policy engine** (DR-008); so a permit decision is a gate verdict in disguise, but on a **distinct axis** (the "may" axis) from the `gate.*` verdict facts (the "did/proved" axis), keyed on the **action request**, not the gate-run.

The verdict contract maps to authorization with **zero new vocabulary**, honoring I6 — verdict is carried by the **subject itself** (as `gate.*` does), never a bare boolean, and `inconclusive` is **never coerced** (DR-008; design §4):

| Verdict | Permit subject | Meaning |
|---|---|---|
| `pass` | `permit.granted` | **allow** |
| `fail` | `permit.denied` | **deny** |
| `inconclusive` | `permit.escalated` | **escalate to a human** (routed to a client, never auto-resolved) |

There is deliberately **no `permit.allowed` synonym** and no subject that coerces an inconclusive to allow (I6). All four subjects fold into materialized state (no consumer-less subjects — DR-006 precedent): a per-run permit **ledger** (request → decision, correlated) plus per-**session accumulators** (running risk score, cumulative spend, tool-use-vs-intent) that the later contextual/stateful C1/C7 permit-verifiers read (design §7; intel memo 001 C6/C7 sizing note). This session mints the subjects + reducer scaffolding only; the verifiers themselves are SP0–SP4 work.

| Subject | v | Emitter | Semantics |
|---|---|---|---|
| `permit.requested` | 1 | harness PEP via `request_permission` (MCP tool / socket, I5); recorded by the daemon PDP | An agent (via the harness PEP) asked to perform an action. Carries the run, a small inline action/target descriptor, and a CAS ref for any large action context (args/diffs/file bytes) — never inline (I2). The `correlation` opens the request→decision chain each decision fact points back at. |
| `permit.granted` | 1 | permit gate engine (PDP; DR-008) | Decision = **allow** (`pass`). Carries `policy_ref` (the deciding permit-verifier/policy) and `evidence_ref` (CAS) so `gate why` / `gate_explain` can explain the grant (I6 interrogability). |
| `permit.denied` | 1 | permit gate engine (PDP; DR-008) | Decision = **deny** (`fail`). Carries the deciding `policy_ref` + `evidence_ref` (CAS) and a short `reason` so a blocked agent can always read *why* (I6). |
| `permit.escalated` | 1 | permit gate engine (PDP; DR-008) | Decision = **inconclusive → routed to a human** (`ask`). Never coerced to allow (I6, DR-008 §4). Carries `policy_ref` + `evidence_ref` (CAS) so the human sees the deciding policy and evidence. |
| `permit.resolved` | 1 | daemon single writer, emitted directly by `call_resolve_permit` (a pure fact emit — no `McpSubstrate` method; a resolution performs no side effect, unlike `kill_run`'s reaper seam) behind the operator-only badge door (DR-033 §Design, corrected by erratum; operator-client slice 2) — **emitter is slice-2 implementer scope** | A human operator **resolved a previously-escalated permit** (`decision = allow \| deny`) via the `resolve_permit` MCP tool. NOT a PDP verdict — the recorded human OVERRIDE the PDP later APPLIES on the agent's next ask for the same action `(run, tool, action/target)`, emitting the corresponding `permit.granted`/`permit.denied` citing this resolution (`resolved_from`) as authority (I6; DR-033 §Decision 1). Never a silent coercion of `permit.escalated` — a recorded, attributable human grant. Carries the escalated `request_id` (audit correlation), the reused `action`/`target` descriptor (the next-ask match key), the operator `reason?`, the loggable `operator_badge_id?` (never the token, §12/I2), an optional `ttl_ms?` — a DURATION relative to this fact's own envelope ULID after which the PDP skips the resolution and the next ask re-escalates (DR-035; log-derived expiry, no wall-clock, absent = permanent), and an optional `scope?: "run_tool"` — an opt-in single-axis wildcard that widens the match from the exact `(run, tool, action/target)` to "any action on this `(run, tool)`" (DR-035 §Decision 2; absent = exact request-scoped match; a broad `scope` obligates a `ttl_ms`, enforced at the `resolve_permit` tool boundary). |
| `permit.delegated` | 1 | daemon PDP, when a lead agent's badge is attenuated for a sub-agent spawn (DR-017 §Decision 2; SP4b macaroon-attenuated delegation) | A capability was **narrowed and handed off**: a lead agent's macaroon badge was attenuated to mint a child badge for a sub-spawn. The attenuation is a **real `Macaroon::attenuate` at the daemon boundary** (DR-018 §Decision (b)) — the daemon calls `attenuate`, not `mint`, so a narrowing caveat **is** appended and the running MAC sig **is** re-keyed without the root key (offline-of-the-root-key: the running sig, not the root key, is the next hop's key — the delegation property, design §4). What is **deferred to a later SP4 sub-slice** (DR-018 §Decision (b)) is the *holder-offline process boundary* — a lead-agent process performing the attenuation itself with no daemon involvement; in SP4b's shipped scope the single daemon holds the base badge and attenuates on the sub-spawn path, so this fact must not be read as holder-offline delegation. A **new event in time** on the `permit.*` "may" axis (contrast the SP4a spawn-time `agent.spawned.role?` field) — it earns a row because the capability chain is a durable fact that must replay (I3). Carries the parent/child `badge_id`s (loggable, never the token) and the added caveats. Folds onto the run's dossier so the delegation chain is queryable/replayable. |

### run

The **run-intent axis** (DR-010, ACCEPTED 2026-07-18; design `docs/design/intent-lock.md`). A run's *initiating intent* and the intent-derived least-privilege tool set the future `intent-lock` permit-verifier (SP-intent) enforces. Keyed on the `run` RunId ULID — the same key discipline `agent.*`/`permit.*` fold on — so the intent folds onto the run's dossier (`AgentRunState`), not a new entity. This is a **distinct axis** from `agent.spawned.allowed_tools?`: that records the *composed harness allowlist* (what the agent was configured with — a governance-composition fact); `run.intent.declared.allowed_tools` records the *intent-derived least-privilege set* the verifier checks a requested tool against (DR-010 §4). A distinct axis warrants a distinct subject (DR-006 precedent: `integrity.*` not folded into `gate.*`). The `run` noun is new; the run's *lifecycle* facts remain grandfathered under `agent.*`, and `run.intent.declared` mints the run-intent axis rather than fracturing it — the subject folds onto the existing `AgentRunState` keyed on `run`, so the two nouns describe one entity, one dossier. This session mints the subject + its folding reducer only; the `intent-lock` verifier is SP-intent (oracle-first, next slice).

| Subject | v | Emitter | Semantics |
|---|---|---|---|
| `run.intent.declared` | 1 | the run/spawn intent-declaration path (SP-intent; project spec `[gates.permit]` / agent spec, design §3(a)) — **minted by SP-intent, not this session** | A run declared its initiating intent and the intent-scoped tool set the `intent-lock` verifier enforces (least-privilege in time). The intent text is a CAS ref (never inline; I2); the allowed-tool set is a short string list. Folds to a rebuild-stable per-run intent state (`AgentRunState.intent`) the verifier reads as a pinned input (I3; DR-010 §4). |

### egress

The **egress-mediation axis** (DR-029, ACCEPTED 2026-07-21; the paired warden mint DR-029 §Decision 6 required, replacing the DR-025/026/028 PLACEHOLDER strings). This noun carries the C3 network-confinement facts a governed run produces: the composed **run-posture** fact (confined + how the network is confined) and the per-connection **denied** fact. The sandbox posture is a **field** on the posture fact (`sandbox: "available" | "unavailable"`), NOT a parallel `sandbox.*` subject: `compose.rs::compose_degrade` decides ONE composed state (the product of C3a's sandbox availability and c3bc's egress availability), so the single decision rides a single fact — splitting it across `sandbox.*` + `egress.*` would fracture one decision into two subjects. This mirrors how `pep?`/`role?` (spawn-time postures) ride as fields on `agent.spawned` rather than minting their own subjects.

Keyed on the `run` RunId ULID — the same key discipline `agent.*`/`permit.*`/`run.*`/`action.*` fold on — so these facts fold onto the run's dossier (`AgentRunState`), NOT a new entity. The `egress` noun is **new**, minted per the DR-006/DR-008/DR-010/DR-021 "distinct axis → distinct noun" precedent. The degrade contract is **CLOSED and loud** (DR-026 §Decision, I6): an unavailable backend NEVER opens unmediated egress and NEVER injects — it keeps the sealed netns and announces `egress.unavailable`; an unsandboxed spawn declares egress un-enforceable rather than silently claiming mediation it lacks (the overclaim the threat model forbids). This session mints the subjects + payload schemas + the reducer-arm obligation; the emitter swap (replacing the placeholders in `compose.rs`/`egress.rs`/`runs.rs`) and the fold are the C3 implementer's `c3-egress-fold` work (oracle-first), NOT written here.

|---|---|---|---|
| `egress.mediated` | 1 | `rezidnt-run` composed spawn path (`compose.rs::degrade_fact`, `ComposedDegrade::Mediated`) — **emitter swap is C3 implementer scope** | The composed run is confined AND egress-mediated: pasta-outer sealed the userspace netns and bwrap confined the filesystem on the SAME shared netns, so the run's sole route out is the folded proxy (DR-028 §Decision 1). The **enforcing** posture, not a degrade — its subject carries no `*.unavailable` marker. Replaces the `sandbox.mediated` placeholder. |

### credential

The **brokered-credential axis** (DR-029, ACCEPTED 2026-07-21; paired mint, DR-029 §Decision 6, replacing the DR-026 PLACEHOLDER). A distinct noun from `egress` because a brokered secret is its own entity — a `secret_ref` — whose lifecycle (injected upstream / dropped-because-unresolvable) is a distinct axis, and whose **leak-discipline is a property of this noun specifically**: every `credential.*` fact carries ONLY the `secret_ref` (a label/hash), the destination, and a `policy_ref` anchor — **NEVER the secret value** (I2/I3; DR-026 crit 5). The value is redacted at the type level in `rezidnt_run::egress::BrokeredSecret` (`Debug`/`Display` print `"<redacted>"`; the value is reachable only through the single audited `expose()` on the upstream-write path), so a value literally cannot ride these facts: the fact constructors take a `&BrokeredSecret` and lift only `secret_ref()`, never `expose()`.

Keyed on the `run` RunId ULID (same key discipline as `egress.*`/`permit.*`) — folds onto the run's dossier (`AgentRunState`), not a new entity. The `credential` noun is **new**, minted per the same distinct-axis-→-distinct-noun precedent. This session mints the subjects + payload schemas + the reducer-arm obligation; the emitter swap (`egress.rs::injected_fact` + the drop path DR-029 §Decision 2 introduces) and the fold are C3 implementer scope, NOT written here.

| Subject | v | Emitter | Semantics |
|---|---|---|---|
| `credential.injected` | 1 | the egress dataplane upstream-injection point (`egress.rs::injected_fact`, post-TLS-termination on the plaintext the agent never sees) — **emitter swap is C3 implementer scope** | A brokered secret was injected into the UPSTREAM request on an approved mediated egress (DR-026 crit 4). The PDP records THAT a secret was injected and WHICH (its `secret_ref`) + WHERE (`dest`) + under which policy (`policy_ref`) — **never the value** (the by-reference contract; the value is exposed only at this injection point, never on the fabric). Replaces the `credential.injected` placeholder. |
| `credential.dropped` | 1 | the daemon-side fold / `SecretSource` resolution path (DR-029 §Decision 2 — `secret_ref` the source cannot resolve) — **emitter minted by the C3 implementer, not this session** | A `secret_ref` declared in the `[egress.secrets]` map could NOT be resolved by the `SecretSource`, so that host→secret mapping was DROPPED with a loud fact — never a fake/empty secret, never an injection of the empty string (the honest floor, DR-029 §Decision 2). The run still proceeds mediated for hosts whose secrets resolved. A `credential.*` variant (not an `egress.*` degrade) because the dropped thing is a **credential** identified by its `secret_ref`; the fact carries the unresolvable `secret_ref` + the `dest` it was mapped to + a `reason` so *why uninjected* is interrogable (I6). Carries no value (there is none — that is the point). |

## Payload schemas — v1 baselines

Ratified per-subject payload shapes. A field marked `?` is optional and may be absent; readers tolerate unknown fields (additive evolution, doc §5). Types map JSON onto the rezidnt-types Rust shapes; `CasRef` is `{hash: blake3 hex string, bytes: u64, mime: string}`. Subjects not listed here have no ratified payload schema yet — their shape is proposal-stage until a warden session ratifies it. These baselines define v = 1; a breaking change to any of them mints v+1 per the change discipline above.

### S1 set (ratified 2026-07-16)

**`workspace.opened` v1** — the envelope `workspace` id is the entity key (reducers fold on it); the payload carries human-facing identity.
- `name: string` — workspace name from the project spec.
- `root: string` — canonicalized absolute path of the workspace root.

**`workspace.spec.applied` v1**
- `spec_ref?: CasRef` — the applied spec file as persisted to the CAS; optional because an open may apply an inline/default spec with no persisted blob.
- `agents: [string]` — spec agent names configured by the applied spec.

**`worktree.allocated` v1**
- `path: string` — canonicalized worktree path; exists on disk at emission time.
- `branch?: string` — branch checked out in the worktree, when one was requested.
- `allocator: "rezidnt"` — sole-allocator model (DR-001). The value `"human"` is reserved for out-of-band observation and is never emitted by rezidnt on this subject.

**`agent.spawned` v1**
- `run: string` — RunId ULID; the key every `agent.*` fact carries.
- `agent: string` — spec agent name.
- `harness: string` — harness identifier (e.g. `claude-code`).
- `harness_version?: string` — as probed at spawn; version-gated per adapter.
- `pid?: u32` — OS process id when known at emission.
- `badge_id: string` — loggable badge identifier. NEVER the badge token (doc §12); the token exists only in the spawned environment. For a macaroon (agent) badge, this id is derived from the running MAC sig — `hex(blake3(sig)[..8])` (DR-018 §Decision (a)), not `blake3(identifier)[..8]`; see the `permit.delegated` id fields for the rationale (a true `attenuate` preserves the identifier, so a sig-derived id is what distinguishes parent from child edge ids). This is distinct from the DR-005 daemon-lifetime **operator** badge (§ badge / doc §12), which is not a macaroon and whose id derivation DR-018 does not touch.
- `idempotency_key?: string` — caller-supplied spawn idempotency key, recorded on the fact so the key→run map is log-derivable (I3; S3-T1 remediation, pre-S4). Present iff the spawning call supplied one: MCP `spawn_agent` requires a key (`SpawnAgentArgs.idempotency_key`), other spawn paths carry none — optional is the honest shape, never synthesized for keyless spawns. Constraints: non-empty, ≤ 256 bytes UTF-8 (DEFAULT cap; a key is a short opaque token, trivially inside I2). Opaque to the daemon beyond byte equality. Dedup scope is per workspace — the envelope `workspace` id paired with the key: a spawn request bearing a key already recorded on an `agent.spawned` fact for that workspace answers with that fact's `run` and emits nothing new, including across daemon restart. Emitter obligation: a keyed spawn fact MUST set the envelope `workspace` field, or the rebuilt map has no scope. **Added 2026-07-17 (additive; v stays 1).**
- `bare?: bool` — whether the spawn was governed as a *bare* agent (no interactive/permission-prompt affordances): the enforcement decision the `vet` gate's `bare-mode` verifier checked, recorded on the fact so the governance posture is log-derivable (DR-001: enforcement decisions recorded in events; I3). Optional/additive: present on governed spawns that ran through `vet`; absent on legacy or ungoverned spawn paths (never synthesized — absence is honest, not `false`). **Added 2026-07-17 (S4; additive; v stays 1).**
- `allowed_tools?: [string]` — the composed allow-list of tool names the agent was spawned with, as enforced pre-spawn — the permission composition the `vet` gate's `allowed-tools` verifier checked (DR-001: allowedTools recorded in events; I3). Records what was granted, for log-derivable attribution of what the agent *could* do vs. `agent.tool.invoked` facts for what it *did*. Optional/additive: present on governed spawns; absent when no tool policy was composed. A short list of short strings — trivially inside I2; a pathological policy stays a CAS concern only if it ever exceeded the envelope cap (it does not for tool-name lists). **Added 2026-07-17 (S4; additive; v stays 1).**
- `pep?: "enforced"` — **enforcement-mode visibility (DR-014 §Decision 5, Design §6).** Records whether this run had the permit **Policy Enforcement Point** (the `rezidnt permit-hook` `PreToolUse` hook) wired at spawn, so `gate_explain` can honestly distinguish a *mid-run-PEP-enforced* run from an *edge-gated-only* run (I4 degradation honesty). The value is fixed at spawn — opt-in is keyed on the spec declaring a `[gates.permit]` gate (DR-014 §Decision 2; Design §3), and the enforcement mode **never changes after spawn**, so this is a spawn-time property (a field), not a lifecycle event of its own. **Value semantics (DR-012 declared-vs-absent discipline):** `"enforced"` — present iff `SpawnPlan::for_claude_code` wired the PEP (the run's spec declared a `[gates.permit]` gate); mid-run tool calls are intercepted and asked of the daemon PDP. **Absent** — the run was **edge-gated-only**: no `[gates.permit]` gate, or a legacy/ungoverned spawn path that wired no hook; it still gets pre-spawn `vet` + post-hoc `debrief` evidence but **no mid-run interception**. Absence is the honest representation of "no PEP wired" — never synthesized to a `false` / `"unenforced"` value (exact mirror of the `bare?` / `allowed_tools?` absent-is-honest discipline; DR-012 keeps declared-empty distinct from absent, but here there is no "declared-empty" state — the PEP is wired or it is not, so a single present value + honest absence is the correct shape). An **enum string, not a bool**, so a future degraded/partial enforcement mode (e.g. hook present but socket unreachable at spawn) can arrive additively as a new value without a `v+1` (same additive-string discipline as `agent.status.changed` / `gate.inconclusive.reason`); in v1 the only value any emitter produces is `"enforced"`. Independent optional — does not collide with `bare?` / `allowed_tools?` / `harness_version?` / `idempotency_key?` / `badge_id`. **Added 2026-07-18 (DR-014; additive; v stays 1).**
- `role?: string` — **RBAC role — permit input axis (DR-016 §Decision 2, Design §4).** The agent's authorization role, taken verbatim from `AgentSpec.role` (`role: Option<String>`, DR-016 §Decision 2), recorded on the spawn fact so permit policies can key decisions on **role + workspace + action** (permit-engine §7). Injected into `decide_permit`'s content-pinned per-run params (DR-011 §2) as a **new input axis** both native and exec permit verifiers read — no new verifier machinery, a new input field on the existing SP1–SP3 dispatch. Like `pep?`, this is a **spawn-time property of the run** (the role is fixed by the spec at spawn and rides the run's derived state), not a lifecycle event of its own — a field, not a subject. **Value semantics (DR-012 declared-vs-absent discipline):** an **opaque string the policy interprets** (rezidnt mints no role vocabulary — DR-016 does not design the role taxonomy); present iff `AgentSpec.role` declared one. **Absent** — the spec declared **no role**; the run has no RBAC role and a role-keyed policy sees no role to match. Absence is the honest representation of "no role declared" — **never synthesized to a default** like `"contributor"` (exact mirror of the `bare?` / `allowed_tools?` / `pep?` absent-is-honest discipline; DR-012 declared-vs-absent — a role-less agent is not a `"contributor"` agent). A short scalar (a role name is a short opaque token, trivially inside I2 — no CAS ref, no accumulator). Independent optional — does not collide with `pep?` (enforcement *mode*) / `bare?` / `allowed_tools?` (composed tool allowlist) / `harness_version?` / `idempotency_key?` / `badge_id`; `role?` records *who the agent is authorized as*, a distinct axis from all of them. **Added 2026-07-19 (DR-016 SP4a; additive; v stays 1).**
  - Note on `harness_version`: already ratified above as `harness_version?: string` (S1 baseline). The `vet` gate's `pinned-version` verifier reads/enforces it; no new field is minted for version pinning — the existing optional field carries the recorded decision. Cross-referenced here so the governed-spawn set (`bare`, `harness_version`, `allowed_tools`, `pep`, `role`) reads together.
- Interaction check (recorded 2026-07-17; extended 2026-07-18 for `pep`, 2026-07-19 for `role`): the governed-spawn fields (`bare?`, `allowed_tools?`, `harness_version?`, `pep?`, `role?`) are mutually independent optionals and do not collide with `idempotency_key?` (dedup key) or `badge_id` (attribution). The `badge.issued` emit-or-drop and `badge_id`-on-mutation-facts deferrals are untouched and not foreclosed — no field added here occupies a name or scope reserved for those decisions. `pep?` records a *spawn-time enforcement property* (was the PEP wired), a distinct axis from `allowed_tools?` (what tools were composed) — the two answer different `gate_explain` questions and never coincide. `role?` records the *RBAC role the run is authorized as* (a permit input axis, DR-016) — distinct again from `pep?` (enforcement *mode*, not identity) and from `allowed_tools?` (a *tool* allowlist, not a *role*); a policy reads `role` + workspace + action, not the tool set.

**`agent.status.changed` v1** — state delta.
- `run: string`
- `from: string`, `to: string` — run-status vocabulary: `spawning | running | completed | failed | signaled`.

**`agent.completed` v1** — dossier accounting (DR-001).
- `run: string`
- `status: "success" | "error"` — the harness result outcome. This is a distinct vocabulary from the run-status values of `agent.status.changed` (`completed`/`failed`); reducers must not conflate the two.
- `cost: {total_usd: f64, input_tokens: u64, output_tokens: u64}`
- `num_turns: u64`
- `duration_ms: u64`
- `session_id?: string` — harness session id, captured for run checkpointing (`--resume`, DR-001).

**`agent.signaled` v1**
- `run: string`
- `signal: string` — the delivered signal name (e.g. `SIGTERM`, `SIGKILL`).
- `escalation?: "term" | "kill"` — present when the signal came from the reaper's TERM→KILL escalation path, recording the stage; absent for out-of-band signals.
- `operator_badge_id?: string` — **operator attribution (DR-032 §Decision 4/5).** The loggable badge identifier of the human operator who requested this stop via the `kill_run` MCP tool, so a human-initiated kill is interrogable and distinct from a daemon-initiated stop (I6). This is the **DR-005 daemon-lifetime operator badge** id — NOT a macaroon agent badge (DR-032 §Decision 1 rejects the macaroon path for kill: terminating a run is an operator action, not an agent self-action), and NEVER the badge token (§12; the token rides only the 0600 lockfile, never the fabric — I2). Named `operator_badge_id` (not the bare `badge_id` the mutation-request path uses at ~386) to disambiguate the admitted class: only the operator badge authorizes a kill. **Value semantics (DR-012 declared-vs-absent discipline):** present iff the `agent.signaled` fact was emitted through the `kill_run` operator path (the reaper stop driven by an operator's `check_badge`-verified call). **Absent** — a **daemon-initiated stop**: the reaper's own TERM→KILL escalation on a timeout/lifecycle condition, or any out-of-band signal, has no requesting operator. Absence is the honest representation of "not operator-attributed" — **never synthesized** to a sentinel/empty string (exact mirror of the `bare?`/`pep?`/`role?` absent-is-honest discipline; a daemon stop is not an operator stop). A short 8-byte hex loggable prefix (I2-safe; no CAS ref). Independent optional — does not collide with `escalation?` (which records the reaper *stage*, orthogonal to *who requested it*): a daemon escalation carries `escalation` but no `operator_badge_id`; an operator kill that itself runs through the reaper's TERM→KILL may carry BOTH (the operator requested it AND it escalated) — the two answer different `debrief`/`gate why` questions. **Added 2026-07-22 (DR-032; additive; v stays 1).**
- `reason?: string` — **operator-supplied kill reason (DR-032 §Decision 5).** An optional short human-readable reason the operator gave for the stop, carried so `debrief` / `gate why` can show *why* a human killed the run (I6 interrogability). Small inline text — a reason is a short sentence, trivially inside I2; a pathological over-long reason is the caller's concern to bound (DEFAULT cap ≤ 1 KiB, keeping envelope headroom under the 32 KiB I2 hard cap), never a CAS ref for a normal reason. **Value semantics:** present iff the operator supplied one on the `kill_run` call; **absent** on a daemon-initiated stop (no operator, no reason) AND on an operator kill that gave no reason (a reason is optional even on the operator path — absence is honest "no reason given", never synthesized to `""`). Independent optional; folds/reads alongside `operator_badge_id?` but does not require it (an operator MAY be attributed with no reason, or — by construction of the daemon path — neither). **Added 2026-07-22 (DR-032; additive; v stays 1).**

**`agent.tool.invoked` v1** — harness telemetry (DR-001).
- `run: string`
- `tool: string` — tool name as reported by the harness.
- `input_summary?: string` — truncated human-readable summary of the tool input; bulk input goes to the CAS and rides `artifact.captured`, never inline (I2).

**`agent.message` v1** — harness telemetry (DR-001). Carries exactly one of `text` / `ref`.
- `run: string`
- `role: "assistant"` — fixed in v1; other roles arrive additively.
- `text?: string` — inline only when ≤ 8 KiB (DEFAULT cap; keeps envelope headroom under the 32 KiB I2 hard cap).
- `ref?: CasRef` — bulk message body persisted to the CAS.

**`artifact.captured` v1**
- `ref: CasRef` — carries `hash`, `bytes`, `mime`; the v0 sketch's top-level `mime`/`bytes` live inside the ref (no duplicated fields).
- `provenance: {run?: string, kind: string, chunk?: u64}` — `run` when the artifact belongs to an agent run; `kind` names the capture class (e.g. `capture-chunk`, `diff`, `gate-evidence`); `chunk` is the 0-based ordinal within the run's capture stream, present iff `kind = "capture-chunk"`. DR-001 chunked run output rides this subject via ref-only manifest facts (`rezidnt-run` `ManifestEntry {run, chunk, ref}`); whether high-rate capture chunks deserve a dedicated subject is an open question flagged for `/dr` — not minted here.

### S2 set (ratified 2026-07-17)

**`worktree.observed` v1** — out-of-band guard fact (DR-001); registers the tree so re-observation stays silent.
- `path: string` — canonicalized path of the observed tree; the registry key it occupies.
- `allocator: "human"` — fixed in v1: observation is by definition out-of-band. The value `"rezidnt"` is never emitted on this subject (exact mirror of the `worktree.allocated` reservation).
- `branch?: string` — branch checked out in the observed tree when the watcher can read one; absent for detached human trees. Additive; family coherence with `worktree.allocated`.

**`worktree.conflict` v1** — emitted instead of double-tracking. **Emission is at-least-once, not exactly-once (honest wording, S2 re-debrief T2).** The adapter emits the conflict fact and *then* persists the `conflicted` dedup mark (`crates/rezidnt-adapters/git/src/lib.rs`: in `observe`/`allocate`/`discover` the sequence is set-flag → `emit("worktree.conflict")` → `persist_registry`). In the crash-free path this is exactly-once: the persisted mark suppresses re-emission on the next collision, forever. But a crash *between* the emit and the registry persist loses the mark, so on restart the next claim on the same path re-emits — hence at-least-once across a crash. Reducer obligation (I3): each logged fact is folded, and a downstream consumer treats `worktree.conflict` as idempotent per contested path (dedup on `path`), because the log is the source of truth and may honestly carry a duplicate the crash produced. This describes the code's real semantics; it is a wording correction, not a design change.
- `path: string` — the contested **canonicalized registry key** (the path as registered, not the colliding spelling).
- `claimed_path?: string` — the colliding claim as observed, pre-canonicalization, present when it differs from `path`. Triage evidence: names the spelling that collided without a registry read.
- `holder?: "rezidnt" | "human"` — allocator recorded on the standing registry entry at collision time, making the fact self-contained for triage. (Named `holder`, not `allocator`: on this subject "allocator" would be ambiguous between the entry's holder and the out-of-band claimant.)

**`worktree.released` v1**
- `path: string` — canonicalized registry key; byte-identical to the spelling the allocation minted (both are canonicalized, so the strings agree — the release fact closes exactly the entry the allocated fact opened).
- `branch?: string` — branch the worktree carried, when it had one. Additive; lets a consumer of the release fact alone know what went away, family coherence with `worktree.allocated`.

**`diff.ready` v1** — I2 contract: the diff summary is a CAS ref, never inline diff bytes.
- `worktree: string` — canonicalized path of the worktree the diff concerns (the same registry key `worktree.allocated` minted).
- `diff: CasRef` — the diff summary persisted to the CAS; resolvable at emission time. `mime` is `text/x-diff` (DEFAULT, not load-bearing); a real tree change yields a non-empty summary (`bytes > 0`).

### S3 set (ratified 2026-07-17)

Strict supersets of the S3 oracle pins (`rezidnt-mcp/tests/gate_explain.rs`, `bins/rezidentd/tests/mcp_http.rs`, `spec/fixtures/s3_gate_forced_failure.jsonl`, `s3_gate_inconclusive.jsonl`). The verdict is carried by the **subject itself** (`gate.failed` → `fail`) — never a bare boolean field, and `inconclusive` is never coerced (I6). The full verifier engine is S4; these baselines cover exactly what S3's forced-failure / `gate_explain` slice emits and interrogates, and S4 extends them additively (e.g. network-opt-in recording, `cost_ms`). `gate.passed` has no S3 emitter or pin and its baseline is deliberately **not** ratified here — S4 scope.

**`gate.entered` v1** — the envelope `correlation` groups the gate run (doc §5); `causation` on subsequent verdict facts points back at this fact.
- `run: string` — RunId ULID; the key every gate fact carries.
- `gate: string` — the named policy point (`vet`, `pre_merge`, `post_run`); a string, not a closed enum — gate defs are named policy points (doc §8).

**`gate.failed` v1**
- `run: string`, `gate: string` — as `gate.entered`.
- `verifier: string` — the **failing** verifier's name (doc §8; exactly what `gate_explain` must return).
- `evidence: [CasRef]` — evidence blobs live in the CAS; the fact carries refs only (I2, doc §8 BINDING).
- `inputs: object` — the exact verifier input document, recorded **verbatim** (doc §8 stdin contract: `{gate, refs, params, timeout_ms}`; `refs` values are `cas:blake3:<hex>` strings — inputs pinned by content hash, per the determinism BINDING). Deliberately an opaque-but-recorded object, not a decomposed schema: `gate_explain` returns it byte-for-byte from the log (I6 interrogability), and the S4 engine widens the document additively without a payload break here.

**`gate.inconclusive` v1** — same shape as `gate.failed` plus `reason`; routed to a human, never coerced to `pass` (I6).
- `run`, `gate`, `verifier`, `evidence: [CasRef]`, `inputs` — exactly as `gate.failed`.
- `reason: string` — v1 vocabulary from the §8 causes: `timeout | nonzero_exit | malformed_output | could_not_run`; new causes arrive additively as strings.
  - `could_not_run` — the verifier program could not be executed (spawn/exec failure, e.g. argv[0] points at a nonexistent binary) so **nothing ran**; distinct from `malformed_output`, which is a program that **did** run but emitted unparseable output. The verdict stays `Inconclusive` — a verifier that cannot run is never coerced to `pass` (I6). Wire form is the snake_case string `"could_not_run"` (pinned by `crates/rezidnt-gate/tests/exec_could_not_run.rs`). **Added 2026-07-17 (additive string under the existing "new causes arrive additively" rule; `gate.inconclusive` stays `v = 1`).**

**`gate.explained` v1** — interrogations are facts too. The explanation content (verifier, evidence, inputs) is **derived** from the verdict fact already on the log (I3) and is not duplicated into this payload.
- `run: string` — the interrogated run; the pinned minimum is this field alone.
- `gate?: string`, `verdict?: "pass" | "fail" | "inconclusive"` — optional self-contained triage context; when present, `verdict` is the recorded verdict verbatim (never coerced, I6).

### S4 set (ratified 2026-07-17)

Strict supersets of the S4 oracle pins: `crates/rezidnt-gate/tests/`, `crates/rezidnt-state/tests/s4_gates.rs`, `bins/rezidentd/tests/{vet_gate.rs,golden_path.rs}`, and the S4 fixtures `spec/fixtures/s4_{verified_run,vet_refusal,replay_verified,replay_divergence_alarm}*`. Additive-only; every subject stays `v = 1`. The `inputs` document is recorded **verbatim** per the §8 stdin contract and is measured at ~182 bytes for the largest S4 pin (a `pre_merge` diff-scope record with a glob `allow` list); a full 3-verifier `gate.passed` payload is ~838 bytes — three orders of magnitude under the 32 KiB I2 hard cap. Tool-name lists and glob params are short strings; larger content is already a CAS ref by the §8 contract, so no S4 payload approaches the cap.

**`gate.passed` v1** — the S3 baseline deliberately left this unratified ("until an emitter exists"); the S4 engine is that emitter. Now ratified.
- `run: string`, `gate: string` — as `gate.entered`.
- `verifiers: [VerifierRecord]` — one record **per verifier** that ran on the gate, in execution order. Replay re-executes each verifier against its own recorded inputs, and the slice exit requires per-verifier recorded cost — so the evidence is carried per verifier, not flattened on the event. Each `VerifierRecord` is:
  - `verifier: string` — the verifier's name (§8).
  - `cost_ms: u64` — wall-clock cost of this verifier's execution (the §8 stdout `cost_ms`); the "recorded cost" the golden-path exit asserts.
  - `evidence: [CasRef]` — this verifier's evidence blobs as CAS refs (I2); empty when the verifier emitted none. The reducer flattens all records' evidence hashes in order into `GateState.evidence`.
  - `inputs: object` — the exact per-verifier §8 stdin document, recorded verbatim (`{gate, refs, params, timeout_ms}`; `refs` values are `cas:blake3:<hex>` — inputs pinned by content hash, determinism BINDING). Same opaque-but-recorded discipline as `gate.failed.inputs`; widened additively by the engine without a payload break.
- **Asymmetry with `gate.failed` v1 (rationale, ratified):** `gate.failed` carries exactly ONE verifier record (flat: `verifier`, `evidence`, `inputs` at payload top level) — the *first* verifier to fail short-circuits the gate, so there is a single failing verifier and no further verifiers ran. `gate.passed` carries ALL verifier records because a pass means every verifier ran to completion and each contributes replayable evidence + recorded cost. The shapes are deliberately different (single-failing vs. full-pass-evidence), NOT a nested-vs-flat accident: the pins fix both (`vet_gate.rs` asserts a `verifiers` array on passed and a top-level `verifier` on failed; `s4_gates.rs` folds both). This asymmetry is intentional and load-bearing; it is not a candidate for later "harmonization" without a `v+1` on both subjects.

**`diff.merged` v1** — NEW subject (golden-path merge fact); closes the worktree lifecycle after a verified `pre_merge` pass.
- `run: string` — RunId ULID; ties the merge to the agent run (the golden-path exit asserts `diff.merged.payload.run == run_id`).
- `worktree: string` — canonicalized worktree path; the same registry key `worktree.allocated`/`diff.ready` minted. The reducer folds `status = "merged"` here, inserting the entry even if never allocated (the log is truth, I3 — `s4_gates.rs::diff_merged_marks_the_worktree`).
- `diff: CasRef` — the merged diff summary as a CAS ref (I2); the reducer pins `last_diff = Some(diff.hash)`.
- **New-subject checklist (recorded):** *grammar* — `diff.merged` parses `noun.verb`, past tense for a fact (conformant; unlike grandfathered `diff.ready`). *family coherence* — sits in the `diff` section with `diff.ready`; shares the `worktree` + `diff: CasRef` shape of `diff.ready` and adds `run` (the merge is attributable to a run; a diff-summary is not). *emitter* — git adapter (RepoSubstrate), the S4 golden-path emitter. *reducer consumer exists* — yes: `rezidnt-state` S4 reducer scaffold folds it (`s4_gates.rs` test `diff_merged_marks_the_worktree`; `s4_verified_run.expected.json` counts it). *not a duplicate of `merge.completed`* — `merge.completed` (v0, `merge` section) is the raw merge-mutation fact; `diff.merged` is the worktree-lifecycle-closing fact keyed on the run and folded by the S4 reducer. They coexist; `diff.merged` is not a rename or synonym (distinct key set, distinct consumer).

### permit set (ratified 2026-07-17 — DR-008 / DR-009)

The pre-hoc authorization subjects. Minted at `v = 1`. Verdict is carried by the **subject** (`permit.granted`/`permit.denied`/`permit.escalated`), never a bare boolean; `inconclusive → permit.escalated` is never coerced (I6, DR-008 §4). Large action context (command args, file bytes, diffs) is **always a CAS ref, never inline** (I2). Payloads are sized to **support contextual/stateful downstream decisions** (intel memo 001 C6/C7): decision facts carry `policy_ref` + `evidence_ref` so the interrogation (`gate why` / `gate_explain`) and the per-session accumulators (risk/spend/intent) can be derived from the log alone (I3). This is subject + reducer scaffolding; the C1/C7 verifiers are SP0–SP4 work, not minted here.

Correlation model: `permit.requested` opens the chain via the envelope `correlation` (doc §5); each decision fact carries `causation` = the request event id and repeats the request's `correlation`, so the ledger folds request→decision from the envelope without a payload back-pointer (mirrors the `gate.entered`→verdict correlation model in the S3 set). A `request_id` field is additionally carried on the payload as the explicit, log-derivable ledger key (the request's own ULID text) so the ledger rebuilds even if a decision arrives with the envelope back-pointers absent — honest key, never synthesized.

**`permit.requested` v1** — the request fact. Small descriptor inline; large context is a CAS ref (I2).
- `run: string` — RunId ULID; the key every `permit.*` fact carries (the agent run asking to act). Ties a decision back to its run for the per-session accumulators.
- `request_id: string` — this request's own ULID text; the ledger key the decision facts reference. Log-derivable, never synthesized.
- `action: string` — the action kind, a short closed-ish vocabulary string (v1: `"tool.invoke"`; new kinds arrive additively as strings, same discipline as `gate.inconclusive.reason`). What the agent wants to do.
- `target: object` — a **small inline descriptor** of the action target: a `tool?: string` (tool name, e.g. `"Bash"`), and other short scalar hints. Bounded to short strings — a descriptor, not the payload. Any large context (command args, file contents, diffs) does **not** live here.
- `context_ref?: CasRef` — the large action context (full argv, file bytes, diff) persisted to the CAS; carried as a ref, never inline (I2). Optional: absent when the action has no bulk context (a descriptor-only request).
- `badge_id?: string` — loggable badge identifier of the caller whose capability authorizes the eventual mutation (doc §12; DR-005). NEVER the badge token. Optional/additive: absent on unbadged/read-class request paths; present when the request rode a badge (the request's *result* authorizes a mutation, so the badge is the caller's identity — design §5).

**`permit.granted` v1** — decision = allow (`pass`). Interrogable: the deciding policy + evidence are on the fact (I6).
- `run: string` — as `permit.requested`; the run the decision is for (drives the per-session accumulators).
- `request_id: string` — the `permit.requested.request_id` this decision answers; the ledger correlation key.
- `policy_ref: CasRef` — the deciding permit-verifier/policy, content-addressed: which policy allowed this action, so `gate why` / `gate_explain` returns it (I6). A CAS ref (the policy document may be a Cedar/Rego file or a native verifier's recorded params — bytes go to the CAS, never inline; I2).
- `evidence_ref?: CasRef` — the decision evidence (the verifier's recorded findings/inputs) persisted to the CAS; ref only (I2, mirrors `gate.*` evidence). Optional: absent when an allow produced no evidence blob (a trivially-permitted action).
- `cost_ms?: u64` — wall-clock cost of the decision (the §8 stdout `cost_ms`); additive, absent when not measured. Supports the latency risk-tracking of design §10.2. **Intentionally recorded-only (SP5 close-out):** carried on the decision fact for per-decision latency/audit fidelity — unlike `risk_delta?`, it has **no folding accumulator** (`rezidnt-state` deliberately does not sum it into `PermitAccumulators`; there is no `cumulative_cost_ms`). Reading it back is a per-fact, log-derivable read (I3), never a materialized running total. A future accumulator, if ever wanted, would be its own change (a new `PermitAccumulators` field + reducer arm) — not a payload break (`v` stays 1; the field is unchanged). `permit.denied`/`permit.escalated` inherit this via their "exactly as `permit.granted`" `cost_ms?` line.
- Accumulator inputs (additive, all optional — sized per intel memo 001 C6/C7 so the reducer can fold running per-session state without a `v+1`):
  - `risk_delta?: f64` — the incremental risk this action contributes to the running per-session risk score (C6 contextual policy). **This is the C6 fold source (DR-024, 2026-07-20).** DR-024 confirms `risk_delta?`'s placement on `permit.granted` as correct, not incidental (Q2): risk is deterministically COMPUTED pre-action from the request axis (`tool`/`paths`/`role`) by the `RiskCap` native, so it legitimately rides the permit fact — NOT the dishonest-if-denied pre-action *estimate* that spend was (why `spend_delta_usd?` was retired, DR-021 B2). PRODUCED only here: the emit site stamps it into the existing `DecisionDeltas` carrier via the shared deterministic `risk_score(request_axis, table)` pure fn that `RiskCap::verify` also calls for its soft/hard verdict, so the stamped delta and the verdict's evidenced score cannot diverge (contract-free producer seam — `VerifierOutput`/`PermitOutcome` UNCHANGED, DR-024 Q5→option-iii). FOLDED GRANTED-only (DR-024 Q3): the reducer sums it into `PermitAccumulators.risk_score` on the `"granted"` arm ONLY — a denied/escalated action never ran, so it folds ZERO risk (the I3 honesty property, mirror of DR-021 B2 spend). Same grain/shape as the retired `spend_delta_usd?` and the recorded-only `cost_ms?` (a small optional scalar); unlike `cost_ms?` it HAS a folding accumulator. Absent when not scored.
  - **`spend_delta_usd?` RETIRED as the C1 fold source (DR-021 B2, 2026-07-20):** the incremental-spend field is **removed** from this payload. Under DR-021 §Decision B2 the pre-action permit decision folds NO spend (a pre-action estimate on a granted/denied fact is the dishonest-if-summed accumulator DR-021 rejected on I3 grounds); the **measured** spend now rides the post-action `action.metered` subject (see the DR-021 set below), whose reducer arm folds `cumulative_spend_usd`. The field was never emitted by a live producer (the C1 path was inert — `DecisionDeltas::default()` hard-coded, DR-021 §Context), so its removal is not a breaking payload change and `permit.granted` stays `v = 1`. The reducer arm that read it (`crates/rezidnt-state/src/lib.rs:725-726`) moves to `action.metered` in the same commit (implementer obligation — DR-021 set §reducer).
- `resolved_from?: string` — **the deciding authority when this grant was APPLIED from a human resolution (DR-033 §Decision 1, added 2026-07-22; additive optional, `v` stays 1).** The `request_id` of the `permit.resolved` fact whose human `decision = "allow"` the PDP applied on this ask (see the `permit.resolved` baseline in the **DR-033 set** below). Present iff `decide_permit`'s pre-verifier ledger-check matched a prior resolution for this action `(run, tool, action/target)` and emitted this grant *from* it rather than from a verifier verdict; **absent** on an ordinary verifier-decided grant. Makes "granted via human resolution X" a structured, log-derivable read (I6) — the applied decision cites its authority as a fact, not a prose scrape of `reason`. Deliberately NOT overloaded onto `policy_ref`: that is a content-addressed *policy document* (Cedar/Rego/native-verifier params), and a human resolution is neither a policy nor a CAS blob; conflating the two would misreport the deciding authority. Absence is honest "not resolution-applied", never synthesized (DR-012 declared-vs-absent). A short ULID-text scalar (I2-safe; no CAS ref). `permit.denied` inherits the same field via its "exactly as `permit.granted`" line — a `decision = "deny"` resolution applied on the next ask emits a `permit.denied` carrying `resolved_from` = the `permit.resolved.request_id`.

**`permit.denied` v1** — decision = deny (`fail`). Same shape as `permit.granted` plus a `reason` so a blocked agent can always read *why* (I6, DR-005 read-class-interrogation rule).
- `run`, `request_id`, `policy_ref: CasRef`, `evidence_ref?: CasRef`, `cost_ms?`, `resolved_from?: string` — exactly as `permit.granted` (`resolved_from?` present iff this denial was APPLIED from a human `decision = "deny"` resolution — DR-033 §Decision 1; absent on an ordinary verifier-decided deny).
- `reason: string` — short human-readable denial reason (e.g. `"path outside allowed scope"`); the deny detail the interrogation surfaces. I2: a short diagnostic string; large detail rides `evidence_ref`, never inflated inline.
- **`risk_delta?` RETIRED from this payload (DR-024 Q3/Q5, 2026-07-20)** — removed. DR-024 makes `risk_delta` PRODUCED only on `permit.granted` (Q5: the emit site stamps it there via the shared scorer) and FOLDED only on the granted reducer arm (Q3: GRANTED-only fold). A denied action never ran, so folding its risk is a phantom charge — the exact I3 dishonesty DR-021 B2 refused for spend (and the sharpest case for it, exactly as denied was for `spend_delta_usd?`). The field is therefore DEAD on `permit.denied` — never produced, never folded — and is retired, mirroring the `spend_delta_usd?` retire above. (The earlier "a denied sensitive attempt is itself risk signal" rationale is superseded by DR-024: a denied attempt is real signal, but it is surfaced via the `permit.denied` fact + its `reason`/`policy_ref`/`evidence_ref` and the `denied` counter — NOT by charging phantom risk into the running `risk_score` the next decision reads.) The field was never emitted by a live producer (the C6 path was inert until DR-024 wired it), so removal is not a breaking payload change and `permit.denied` stays `v = 1`.
- **`spend_delta_usd?` RETIRED as the C1 fold source (DR-021 B2, 2026-07-20)** — removed from this payload for the same reason as `permit.granted` (the pre-action decision folds NO spend; measured spend rides `action.metered`). A **denied** action is the sharpest case for B2's honesty: a denied action never runs and must contribute ZERO to `cumulative_spend_usd` (DR-021 acceptance-criteria #5 — no phantom charge on a denied action). The field was never emitted by a live producer; its removal is not a breaking change and `permit.denied` stays `v = 1`.

- `target: object` — the **small inline action-target descriptor, the SAME shape as `permit.requested.target`** (a `tool?: string`, e.g. `"Bash"`, plus other short scalar hints; bounded to short strings — a descriptor, never bulk context, I2). Reusing the exact `permit.requested` field name + shape is load-bearing: the ledger-check matches descriptor-to-descriptor with no translation (do NOT invent a parallel shape — the set already has one). Any large context stays out of this fact (there is none to carry — a resolution answers an already-recorded request whose bulk context rode `permit.requested.context_ref`).
- `decision: "allow" | "deny"` — the human's binding choice, the override the PDP applies on the next ask. **Named `allow`/`deny` (not `granted`/`denied`)** — this is the human's *input verb* (what the operator chose), distinct from the PDP's *outcome subjects* (`permit.granted`/`permit.denied`): the resolution is applied THROUGH the PDP, which then emits the granted/denied outcome fact citing this resolution (`resolved_from`). `allow`/`deny` reads cleanest as a human decision and matches the `ResolvePermitArgs { decision: "allow"|"deny" }` tool schema (DR-033 §Design), keeping the tool input and the fact aligned. A closed two-value enum in v1; a future third value (e.g. a scoped/expiring resolution) would arrive additively as a string (same discipline as `gate.inconclusive.reason`). There is deliberately **no coercion**: `permit.escalated` is never SILENTLY flipped — this fact is the RECORDED override the human explicitly made (I6, DR-033 §Invariants).
- `reason?: string` — optional short operator rationale for the resolution (e.g. `"approved after reviewing the diff"`); small inline text, DEFAULT cap ≤ 1 KiB, far under the 32 KiB I2 hard cap — never a CAS ref for a normal reason (exact mirror of the `agent.signaled.reason?` discipline just minted for kill in slice 1). Present iff the operator supplied one; absence is honest "no reason given", never synthesized to `""` (DR-012 declared-vs-absent).
- `operator_badge_id?: string` — the loggable **DR-005 daemon-lifetime operator badge** id of the human who resolved (the `check_operator_badge`-verified id), so the override is attributable and interrogable (I6). **NEVER the token** (§12/I2; the operator token rides only the 0600 lockfile, never the fabric) — the exact leak-discipline just minted on `agent.signaled.operator_badge_id?` in slice 1 (DR-032). NOT a macaroon agent badge (resolving an escalation is an operator action, not an agent self-action — mirrors DR-033 §Design reusing the DR-032 operator door). Present iff emitted through the `resolve_permit` operator path (which it always is in the shipped scope — the tool verifies the badge before any side effect); optional at the schema level for the same absent-is-honest discipline the other operator-attributed facts follow. A short 8-byte hex loggable prefix (I2-safe; no CAS ref).
- `ttl_ms?: u64` — **the resolution's bounded lifetime as a DURATION relative to its own envelope ULID (DR-035 §Decision 1, added 2026-07-22; additive optional, `v` stays 1).** The number of milliseconds after this `permit.resolved`'s envelope-ULID timestamp for which the resolution stays applicable; past that deadline the PDP skips it and the next ask re-escalates. **Log-derived expiry, no wall-clock (the I3 crux).** The deadline is computed at fold-time as `resolution_envelope_ulid_timestamp + ttl_ms`, and applicability is a comparison against the INCOMING `permit.requested`'s OWN envelope-ULID timestamp — both timestamps are already on the log (every event `id` is a time-ordered ULID, `crates/rezidnt-types/src/lib.rs:173`), so the same log always replays to the same decision; there is NO decision-time `SystemTime::now()` read. Chosen as a DURATION, not an absolute-expiry instant, on purpose: the resolution's own creation time IS its envelope ULID (already present — deliberately NO `created_at` is added, that would be a second competing time anchor), so "valid for N ms after I was recorded" keeps the envelope ULID as the SOLE time anchor (I3) and is the natural operator input ("grant for 1 hour"); an absolute wall-clock instant would force a clock read at resolution-mint time — the exact hazard DR-035 §Decision 1 dissolves — and is rejected for that reason. Encoded as `_ms: u64` to match the existing millisecond-integer duration convention (`permit.granted.cost_ms?: u64`, ~397) and the ULID's millisecond-resolution embedded timestamp; a `u64` ms range is unbounded for any real TTL. **Absent = permanent** — today's DR-033 behavior (a resolution stands until overridden); absence is honest "no expiry", never synthesized to `0` (which would mean *instantly* expired) or a sentinel (DR-012 declared-vs-absent). Interrogable (I6): the deadline being a pure `resolved_ulid + ttl_ms` derivation, `debrief`/`gate_explain` renders "not applied: resolution expired at ULID T → re-escalated" from the fact alone, never a silent proceed. `ttl_ms?` was minted by the FIRST DR-035 sub-slice (`escalation-ttl`); the grant-all wildcard `scope?` marker below (DR-035 §Decision 2) was added by the SECOND sub-slice (`escalation-grant-all`). The two are declared INDEPENDENTLY in this schema but are semantically COUPLED by DR-035 §Decision 3 (a broad `scope` obligates a `ttl_ms` — broad OR permanent, never both); that coupling is enforced at the `resolve_permit` tool boundary, NOT in this subject schema (see the `scope?` bullet).
- `scope?: "run_tool"` — **the resolution's MATCH BREADTH: an opt-in single-axis wildcard that widens the exact request-scoped match (DR-035 §Decision 2, added 2026-07-22; additive optional, `v` stays 1).** Absent (the default) = today's DR-033 exact match: the resolution applies ONLY to the exact `(run, tool, action/target)` it answers. Present with value `"run_tool"` = broaden to "any action on this `(run, tool)`": the resolution matches the CLASS of requests sharing the same `run` and `target.tool`, wildcarding the action/target axis. Encoded as a **closed single-value string enum**, NOT a boolean and NOT an expression string, on purpose: a bare `broad: true` boolean would say "this is broad" without naming WHICH axis is wildcarded, whereas the value token `"run_tool"` names the axes held FIXED (`run` + `tool`) with the action/target axis wildcarded — so the encoding IS the predicate and `gate why`/`debrief` render it verbatim ("granted by broad resolution X matching predicate `any action on (run, tool)`, operator badge_id") with no interpretation (I6). A string (vs. a boolean) also lets a FUTURE second named axis arrive ADDITIVELY as a new enum value (same "new causes arrive additively" discipline as `gate.inconclusive.reason`), which a boolean would foreclose. **An unrestricted predicate language (arbitrary boolean expressions over run/tool/target/role) was REJECTED** — DR-035 §Decision 2: maximally flexible but not cheaply interrogable and hard to bound blast radius; a single named axis the human and `gate why` can both read at a glance is the whole point. **Absent = OMITTED never null** — absence is honest "exact request-scoped match", never synthesized to an `"exact"` sentinel (DR-012 declared-vs-absent); every pre-`escalation-grant-all` fixture/peer parses unchanged and matches exactly as before (additive, rebuild-stable, I3). Interrogable (I6): a broad grant is a RECORDED, attributable, explainable fact — the applied `permit.granted`/`permit.denied` cites the deciding resolution AND, via this field, the predicate it matched; a broadened match is NEVER a silent widening. **Coupling (DR-035 §Decision 3), flagged for the implementer, NOT encoded here:** a resolution carrying `scope = "run_tool"` MUST also carry a `ttl_ms` (broad OR permanent, never both — the broad-and-permanent quadrant is structurally unreachable). That obligation is enforced at the `resolve_permit` tool boundary (badge verified, then `scope`-present-implies-`ttl`-present validated, BEFORE any fact is emitted — no partial state), so the fact log can never carry a broad-and-permanent resolution and the reducer/PDP need no guard against a quadrant that cannot be minted. The subject schema declares the two fields as INDEPENDENT optionals (the ontology does not model cross-field obligations); the coupling lives at the tool, not the taxonomy.
