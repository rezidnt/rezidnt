# Decision Record DR-022 — Benchmark-harness slice: decompose the Phase-2 exit into an in-repo, oracle-first `bench/harness` (three metrics in-repo, precision/recall fenced external)

**Date:** 2026-07-20 · **Status:** ACCEPTED (owner) · **Amends:** §16 (gives the Phase-2 exit milestone — "the benchmark harness runs end-to-end against rezidnt itself," `architecture:320` — its first defined acceptance criteria + exit demo, which it currently lacks; the slice-discipline skill enumerates S0–S5 + the permit SP-arc but no `benchmark` slice) and §15/§17 (records the in-repo/external boundary the milestone collides with). No invariant text is rewritten. **May imply a downstream warden `/subject`** (a `bench.completed` metrics subject vs. an out-of-band report artifact) — flagged here, NOT designed here. · **Cites:** the benchmark seam in `.claude/skills/testing-oracles/SKILL.md:34-36` (harness PUBLIC in `bench/harness`, held-out labeled set PRIVATE); §15 `architecture:302` (four locked metrics); §17 `architecture:330` (the held-out set is "Excluded from this repo permanently"). Omnigent as a benchmark **baseline / optional adapter** (`docs/design/permit-engine.md:9,144`), never read for porting (I8, DR-002). · **Builds on:** S4 complete — the end-to-end golden path already lands on the log (`bins/rezidentd/tests/golden_path.rs:38-157`: open→vet→spawn→diff.ready→pre_merge→merge→diff.merged→debrief with recorded cost, one take); gates at `bins/rezidentd/src/gates.rs`; the cost fields the metrics consume already ship — `agent.completed.cost.total_usd`/tokens (`spec/ontology.md:212-218`), per-verifier `cost_ms` on `gate.passed`, and per-action measured spend on `action.metered` (DR-021 C1, commit 17721bb).

## Context

The Phase-2 exit milestone is stated but never sliced: it has NO acceptance criteria, no oracle, and no crate. `bench/harness` is reserved (`architecture:97-98`) but empty — no crate, no `criterion`/`iai` deps, no orchestration loop, no metrics collation. §15 locks **four** metrics: (1) orchestrated task-completion rate, (2) gate precision/recall against labeled defects, (3) worktree merge success, (4) cost per merged verified diff.

**The crux — a hard architectural boundary.** §17 (`architecture:330`) excludes the held-out labeled case set from this repo **permanently**, and §15 makes the harness public but the labeled set private ("contamination destroys a measuring stick's authority"). Therefore metric **(2)** — precision/recall against labeled defects — **cannot be computed in-repo**: it is structurally fenced behind an external private artifact. Metrics (1)/(3)/(4) **are** computable in-repo from data that already lands on the log: task-completion from the golden-path terminal facts, merge success from `diff.merged`, cost-per-merged-diff off the shipped cost fields (metric 4 is "free from the harness cost fields," §15). The harness orchestrates an **existing** path (S4's `golden_path.rs`); it mints no new golden-path pieces.

This record decides how to decompose the milestone into ONE buildable, oracle-first slice: an in-repo harness collating the three measurable metrics + a seam for the external fourth, honest about the boundary the way DR-021 was honest about lagging enforcement.

**Strongest counterargument (dissent, recorded verbatim per house style):** *"A dogfood benchmark measuring only 3 of 4 locked metrics is a half-measuring-stick. Gate precision/recall is the load-bearing metric — it is the whole gate-quality GTM claim (§15 exists to make 'the category argument ours'). Shipping a harness now that computes completion/merge/cost but returns 'inconclusive' for precision/recall risks a benchmark that LOOKS authoritative — it runs e2e, emits a report, has green criteria — while omitting the one metric that substantiates the differentiation. Better to wait until the private labeled set exists so the harness ships whole, rather than mint a public measuring stick with a hole in the middle that a reader mistakes for a full score."* **Counter to the counter:** the boundary is not a gap this DR chose — §17 fences the labeled set out of the repo **permanently**, so "wait until it's in-repo" describes a state that by ratified design never arrives; the harness and the labeled set live in different places forever. The honest move under I6 is to make the seam explicit and return `inconclusive / no labeled set` rather than fabricate or omit — a missing measurement announces itself instead of hiding. Metrics (1)/(3)/(4) are real, in-repo, replayable today and are worth shipping; the exit demo and criteria below FORBID the "looks authoritative" failure by requiring the report to surface precision/recall as an explicit `inconclusive` with the reason, never a blank or a coerced number. The Phase-2 exit is defined (`architecture:320`) as the harness running e2e against rezidnt itself — that is exactly the three-metric dogfood run; substantiating the GTM precision/recall claim against the private set is a separate external activity, not gated on this slice. **This is the honest limit, disclosed — the owner accepts or rejects the trade at ratification.**

## Decision

- **Scope this slice (IN-REPO):** a public `bench/harness` crate that orchestrates the EXISTING golden path headlessly and end-to-end against a target repo (rezidnt itself = the dogfood run), over a set of in-repo scenario cases / N runs, and collates the **three in-repo-measurable metrics** — task-completion rate, worktree merge success, cost-per-merged-verified-diff — each **derived from the event log**, not from wall-clock side channels. Metric 4 reads the already-shipped cost fields (`agent.completed.cost`, per-verifier `cost_ms`, `action.metered`); no new field is minted.
- **Fence this OUT (structurally external, NOT this slice):** metric (2) gate precision/recall against labeled defects. The harness exposes the **seam** to plug a labeled set in (an interface the external private artifact satisfies), but the labeled data and the precision/recall computation live outside the repo **forever** (§17). Absent the set, the seam returns `inconclusive — no labeled set present`, never a score (I6, mirroring DR-021's disclosed lagging enforcement).
- **Replayability (BINDING to the constitution, not new):** the three in-repo metrics are computed from the folded log so a re-run over the same recorded facts yields the same numbers. The one unavoidable timing input — cost/latency is partly wall-clock (`cost_ms`, `duration_ms`) — is recorded ON the log at production time and read back deterministically by the collator; the collator itself introduces no fresh wall-clock read, so the REPORT is replay-stable even though an individual live run's latency is not. This keeps the metric interrogable: the report can name the exact facts each number was folded from.
- **No bench dependency by default (I7).** A hand-rolled log-replay collator suffices — the harness folds recorded facts and counts; it does not micro-benchmark hot loops. `criterion`/`iai` are **rejected for this slice** (they measure wall-clock throughput of code under test, which is not what these four log-derived metrics need, and each adds dep surface against the one-static-binary posture). If a later slice genuinely needs statistical micro-benchmarking, that is its own DR with its own I7 justification.
- **The metrics-report surface is NOT designed here.** Whether the report is an out-of-band artifact (a file/stdout JSON the harness emits) or gains its own event subject (`bench.completed`, folded like other facts) is a **warden `/subject` question** — flagged, not settled. This record commits the *direction* (three in-repo metrics off the log + an external seam for the fourth); the report's on-log-vs-out-of-band shape is that session's to decide.

## Consequences

- **§16 delta:** the Phase-2 exit milestone gains its first acceptance criteria + exit demo (below), so `/oracle` has something concrete to encode. The milestone is no longer criteria-less prose.
- **§15/§17 delta:** the in-repo/external boundary for metric (2) is now recorded as a slice-level constraint, not just a repo-layout note — precision/recall is structurally external and the harness carries only the seam.
- **Risk-register (§18):** *half-measuring-stick risk (new)* — a public harness that omits precision/recall could be misread as a full score; mitigated by the exit criteria REQUIRING the report to surface precision/recall as an explicit `inconclusive` with reason, and by product copy not claiming a gate-quality benchmark number the in-repo harness cannot produce. *scope-gravity risk (§18, carried)* — a benchmark harness before Phase-2 gates were done would be scope gravity; it is NOT here — S4 is complete and the harness orchestrates the finished path, so this passes the phase-exit-demo test.
- **No shipped test or acceptance criterion is weakened.** This slice ADDS a harness + criteria; it softens no existing gate, verifier, or golden. The precision/recall seam returning `inconclusive` is not a weakened criterion — it is the honest report of a measurement whose input is, by §17, permanently out of repo. The three in-repo metrics are new assertions, not relaxations of old ones.
- **Warden `/subject` (deferred):** a possible `bench.completed` subject vs. an out-of-band report artifact — the warden's call, flagged not designed.

## Acceptance-criteria sketch (what `/oracle` encodes once the benchmark slice starts)

1. The harness runs the golden path **end-to-end, headlessly** against a target repo (rezidnt itself) and emits a machine-readable **metrics report** — no TUI, socket/log + CLI only (I1).
2. The three in-repo metrics — **task-completion rate**, **worktree merge success**, **cost-per-merged-verified-diff** — are computed **from the event log** and are **replay-stable**: folding the same recorded facts twice yields identical numbers; the report can name the facts each number folded from (I3/I6). Cost-per-merged-diff reads only already-shipped fields (`agent.completed.cost`, per-verifier `cost_ms`, `action.metered`) — no new field minted.
3. A **deliberately-failing scenario** (a case whose run does not reach a verified merge) is counted as a **task-completion MISS** — a scored zero — **not** a harness crash or an aborted run.
4. The **precision/recall seam is present** but, with **no private labeled set** supplied, returns **`inconclusive — no labeled set present`** in the report — never a fabricated score, never a blank silently read as zero (I6, never coerce a missing measurement into a pass). When a labeled set IS supplied at the seam, the harness computes precision/recall against it; that path is exercised only by the external private artifact and is not asserted in-repo.
5. No `criterion`/`iai` (or other bench) dependency is added; the collator is hand-rolled log-replay (I7). (If a future slice needs micro-benchmarking, its own DR justifies the dep.)

## Exit demo (Phase-2 exit, made concrete)

`bench/harness` is invoked headlessly against rezidnt itself over the in-repo scenario set; one take, socket/CLI only. It orchestrates the existing golden path (open → vet → spawn → diff.ready → pre_merge → merge → diff.merged → debrief) for each case, then emits ONE metrics report carrying: task-completion rate (with the deliberately-failing scenario scored as a miss), worktree merge success rate, cost per merged verified diff (folded off the log's cost fields), and a precision/recall entry reading `inconclusive — no labeled set present`. Re-running the collator over the same recorded facts reproduces identical metric numbers (replay-stable). **This is the Phase-2 exit: the benchmark harness runs end-to-end against rezidnt itself.**

## What this does NOT decide

- **The metrics-report surface** — out-of-band artifact vs. a `bench.completed` event subject: a warden `/subject` session.
- **The private labeled held-out set** and the precision/recall computation over it — permanently external (§17); only the seam is in-repo.
- **The Omnigent-baseline comparison run** (rezidnt vs Omnigent, `permit-engine.md:144`, DR-002 rule 6 black-box) — a separate activity, not this slice.
- **Any micro-benchmark / statistical-throughput measurement** — not needed by these log-derived metrics; a future DR if ever required.
