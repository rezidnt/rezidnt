# Decision Record DR-035 — Escalation resolution scope & lifetime (TTL/expiry + grant-all-matching predicate)

**Date:** 2026-07-22
**Status:** ACCEPTED
**Amends:** DR-033 §Decision 2 (no-TTL default — adds log-derived expiry) and §Decision 3 (request-scoped default — adds an opt-in broad predicate); §16 (permit PDP decision path — `resolution_for` gains an expiry filter and a widened match); §19 (operator write client — `resolve-permit` gains OPTIONAL ttl/scope args, no new verb). Extends DR-034 (live-unblock inherits both features via `resolution_for`). Reaffirms I1's board proof untouched.

## Context

DR-033 shipped resolve-escalation with two DELIBERATELY-NARROWEST defaults, each deferred as demand-gated: §Decision 2 (no TTL — "a future slice MAY add expiry if measured demand shows it") and §Decision 3 (request-scoped, action-matched — "a 'grant-all-matching' variant is explicitly deferred"). The owner has now named BOTH as the next work (2026-07-22). This record designs them TOGETHER on purpose: they are the two axes of "how much does a single human resolution do" — TTL is its DURATION, grant-all is its BLAST RADIUS — and the real security surface is their PRODUCT. A broad grant that also persists is far riskier than either alone; a DR that split them would let that interaction fall through the crack.

**Deferral 1 — TTL vs DR-033's "no clock dependency" (the I3 crux, engaged head-on).** DR-033 avoided TTL specifically to keep the ledger append-only and replay-deterministic: a wall-clock read at decision time makes the SAME log replay to DIFFERENT decisions at different times, which breaks I3. This record must not reintroduce that. The insight that dissolves the objection: the event envelope `id` is a ULID (`crates/rezidnt-types/src/lib.rs:173`, minted by the monotonic `mint_ulid()` `:189-206`) — time-ordered, with the creation timestamp embedded in every event id. So expiry is evaluated from EVENT TIMESTAMPS already on the log: a `permit.resolved` carries a TTL, and when a LATER `request_permission` folds, the PDP compares the incoming request's OWN envelope timestamp against the resolution's envelope timestamp + TTL. Both timestamps are on the log, so a replay of the same log reaches the SAME decision every time — I3 preserved, no hidden clock. Honest subtlety, stated plainly: "expiry" here means "the next request after the deadline sees it expired" (passive, fold-time), NOT an active timer that fires. A resolution with no subsequent request is simply never consulted — which is fine; nothing was waiting on it.

**Deferral 2 — grant-all vs DR-033's request-scoped default (the blast-radius tension).** DR-033 kept resolutions request-scoped so a human grant for one action can NEVER silently authorize an unrelated action (I6 — a grant is a recorded, SCOPED override). Grant-all broadens the match from exact `(run, tool, action/target)` to a PREDICATE over a class. The tension is blast radius: a broad grant authorizes actions the operator never individually saw. This record decides (a) the predicate language stays MINIMAL and interrogable — a wildcard on ONE axis, not an arbitrary expression engine; (b) a broad grant is STILL a recorded, attributable, interrogable fact — `gate why` must show "granted by broad resolution X, badge_id, matched predicate P"; (c) broad grants are opt-in per-resolution — default stays request-scoped, the operator must explicitly widen.

## Decision

1. **TTL = log-derived expiry from event ULIDs (opt-in, absent = permanent).** `resolve_permit` MAY carry an optional TTL. A resolution is applied only when the incoming `request_permission`'s envelope timestamp is at or before the resolution's envelope timestamp + TTL; past that, the resolution is skipped and the request re-escalates. Both timestamps come from event ULIDs on the log — no decision-time wall-clock read. Absent TTL = today's behavior (permanent until overridden). **Rejected alternative:** wall-clock-at-decision-time (`SystemTime::now()` when deciding) — simpler, but makes the same log replay to different decisions and is exactly what DR-033 refused. Rejected to preserve I3.

2. **Grant-all = a minimal opt-in wildcard predicate on ONE axis (default stays request-scoped).** `resolve_permit` MAY carry an optional broadened scope: a wildcard on the tool/action axis (e.g. "any action on this `(run, tool)`") — NOT an arbitrary boolean expression. Absent = today's exact `(run, tool, action/target)` match. A broad resolution is a recorded, attributable, interrogable fact; the applied grant/deny cites the deciding resolution AND the matched predicate. **Rejected alternative:** an unrestricted predicate language (arbitrary expressions over run/tool/target/role) — maximally flexible but not cheaply interrogable and hard to bound blast radius; rejected for a single-axis wildcard the human and `gate why` can both read at a glance.

3. **The interaction: COUPLE the axes — broad OR permanent, never both.** A broad (grant-all) resolution MUST carry a bounded TTL. You can be broad OR permanent, not both — so the dangerous quadrant (broad AND permanent) is structurally UNREACHABLE, not merely discouraged. This matches the house preference for structural guarantees over advisory ones (cf. DR-031 promoting the board's read-only guard to a permanent invariant). **Rejected alternatives:** (i) both axes independent on any resolution — max flexibility, max risk, permits the worst quadrant; (ii) fully independent but flag "broad + long-lived" in `gate why`/`debrief` — advisory only, leaves the worst case representable. Rejected in favor of making the worst case un-representable.

## Design

- **The interaction (blast radius × duration).** The coupling in Decision 3 is enforced at the tool boundary: `resolve_permit` REFUSES a broad-scope resolution that omits a TTL (badge verified, then argument validation, before any fact is emitted — no partial state). Request-scoped resolutions remain permanent-by-default (unchanged). Thus the fact log can never carry a broad-and-permanent resolution; the reducer and PDP need no guard against a quadrant that cannot be minted.
- **The one function both features modify:** `PermitRunState::resolution_for(action, tool)` — `crates/rezidnt-state/src/lib.rs:482-487` — today `self.resolutions.iter().rev().find(...)` last-matching-wins on exact `(action, target.tool)`. TTL = filter out expired resolutions here (the incoming request's envelope timestamp is threaded in as a new parameter). Grant-all = broaden the `find` predicate here (a wildcard-scoped resolution matches any action on its `(run, tool)`). Both DR-034's live-unblock (`recheck_resolution` in `await_unblock`, `bins/rezidentd/src/main.rs`) and DR-033's PDP path (`apply_folded_resolution` / the pre-verifier ledger-check in `decide_permit`, `crates/rezidnt-mcp/src/lib.rs`) call through `resolution_for`, so BOTH features flow to BOTH paths automatically once this one function honors them.
- **Folded type — additive fields.** `PermitResolution` (`crates/rezidnt-state/src/lib.rs:231-256`) gains two optional fields: a TTL (a duration, or an absolute expiry ULID/timestamp — warden's call) and a scope marker (absent = exact-match, present = the wildcard axis). Both `#[serde(default)]`, additive, rebuild-stable (I3) — old fixtures parse unchanged.
- **Subject — PAIRED warden /subject, flagged NOT minted here.** TTL and scope are additive payload fields on `permit.resolved` v1 (schema in `spec/ontology.md` + `crates/rezidnt-types/src/mcp.rs`), mirroring how DR-033 FLAGGED `permit.resolved` rather than minting it in the DR. The exact field shapes (duration-vs-absolute-expiry, wildcard encoding) are the warden's decision. Consumers: the reducer, `resolution_for`, and `gate why`/`debrief`.
- **Tool — additive OPTIONAL args.** `resolve_permit { run, request_id, decision, reason? }` gains optional `ttl?` and `scope?`. Advertised via `schema_for!(rezidnt_types::mcp::ResolvePermitArgs)` — additive optional args keep §9 no-drift (absent = OMITTED never null; absent = today's permanent, request-scoped behavior). No new verb (I5).

## Invariants

- **I3 — THE crux for TTL.** Expiry MUST be log-derived from event-ULID timestamps, NEVER a decision-time wall-clock read. A replay of the same log must reach identical decisions regardless of when replayed. "Expiry" is a fold-time comparison of two timestamps already on the log (resolution envelope + TTL vs incoming request envelope), so it is a pure fold. Grant-all is I3-neutral: a broader match is still a pure fold, no clock, no hidden state.
- **I6 — interrogability.** An EXPIRED resolution that no longer applies, and a BROAD resolution that grants a class, are both explainable: `gate why`/`debrief` shows "not applied: resolution X expired at ULID T → re-escalated" and "granted by broad resolution X matching predicate P, operator badge_id". A broad or expired grant is never a SILENT coercion — DR-033's I6 posture is preserved and extended, not diluted.
- **I2 — unaffected.** No new coupling; no held connection, no inline data-plane payload. TTL/scope are small control-plane descriptors folded verbatim.
- **I5 — additive optional args** on `resolve_permit`, `schema_for!` no-drift; no new verb.
- **I7 — no new dep.** ULID timestamps are already on every event; the predicate is a minimal single-axis wildcard, not an expression engine.
- **I1 — no board linkage** (`rezidnt-tui`). The board's `crate_has_no_writer_dependency` proof stays untouched.

## Consequences

- **Roadmap.** Opens new slice work on ratify; does NOT auto-advance current-slice (that happens on acceptance). RECOMMENDED sequencing: TWO sub-slices, `escalation-ttl` FIRST, then `escalation-grant-all`. TTL first because the coupling in Decision 3 (broad requires TTL) means grant-all DEPENDS on the TTL machinery existing — building grant-all first would either violate the coupling or ship a broken-until-TTL feature. Standard loop each (DR-paired `/subject` → `/oracle` → implement → `/vet` → `/debrief`); the paired `/subject` for the additive `permit.resolved` fields lands with the first sub-slice. §19's register entry for `resolve-permit` is amended to note the optional ttl/scope args.
- **Risk register.** CLOSES DR-033's two carried-forward "MAY add if demand shows" deferrals. ADDS one honest risk per axis: TTL — a resolution silently stops applying at its deadline (mitigated by I6 explain: "expired at ULID T → re-escalated", never a silent proceed or silent deny); grant-all — blast radius, a broad grant authorizes actions the operator never individually saw (mitigated by default-narrow + explicit-widen + the Decision-3 coupling that forbids broad-and-permanent).
- **Test/criterion honesty.** This WEAKENS no DR-033/DR-034 test and lowers no bar; DR-033's exact-match and DR-034's live-unblock tests stay green (absent ttl/scope = today's behavior, still exercised). New criteria ADDED per axis: (TTL) a resolution past its TTL is NOT applied and the next request re-escalates; replay-determinism of expiry — same log, same decision on re-replay. (grant-all) a broad resolution grants a matching SIBLING action but NOT an out-of-scope one; (coupling) a broad resolution without a TTL is REFUSED at the tool boundary.
- Cross-references: DR-033 (parent — pulls forward and AMENDS both its §Decision 2 and §Decision 3 deferrals), DR-034 (live-unblock inherits both features via `resolution_for`), DR-008/DR-009 (the PDP this extends), DR-006 (single-writer append), DR-011 §3 (empty→escalate honesty — a re-escalation on expiry routes through the same honest path).
