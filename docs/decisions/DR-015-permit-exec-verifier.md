## Context

permit-engine В§11-SP3 acceptance is *"an external policy file decides a permit."* The native permit pack (tool-allowlist, path-scope, spend/rate cap, intent-lock) covers the common cases; DR-008 opened the **bring-your-own-DSL** door вЂ” an OPA/Rego or Cedar policy, or any argv program, plugs in as an exec permit-verifier speaking the В§8 JSON contract. rezidnt dispatches a mature engine; it does not build one.

The seam already exists on two sides and the sketch closes the gap between them (sketch В§2): the `ExecVerifier` argv contract is built (`rezidnt-gate/src/lib.rs:867` вЂ” В§8 stdinв†’stdout, wall-clock `timeout_ms`, nonzero-exitв†’`inconclusive{nonzero_exit}`, unparseableв†’`inconclusive{malformed_output}`, network-off + scrubbed env), and the spec already parses exec permit entries (`VerifierSpec.exec`). The gap: `permit_config_for` FILTERS exec entries to native-only and drops them (`mcp.rs:157`), and `permit::aggregate` cannot carry an exec entry. SP3 un-filters and dispatches вЂ” no new verifier machinery, no new vocabulary.

The load-bearing choice is the sync/async seam (sketch В§3): `ExecVerifier::run` is async; `permit::aggregate` runs today inside the `spawn_blocking` `decide_permit` uses for the native pack + CAS. An exec verifier cannot be `await`ed there. Two options: **(A)** lift permit aggregation to the async layer of `decide_permit` вЂ” natives stay in `spawn_blocking`, exec runs via `await`, the aggregator interleaves both in configured order so first-`Fail`в†’Deny short-circuits across native+exec; or **(B)** `block_on` the exec runner inside the sync aggregate.

**Dissent (recorded verbatim per house style):** two counterarguments were raised. First, latency вЂ” a one-shot exec subprocess per governed tool call (fork+exec+interpreter startup) is far heavier than an in-binary native and sits on the agent's critical path (OPA/Cedar cold `eval` is 10sвЂ“100s ms, at or above the SP2 250 ms hot-path budget, permit-engine В§10.2); (B) is the cheapest diff. Second, **strategic dilution (permit-engine В§10.3, restated verbatim):** SP3 is table-stakes DSL parity, not the evidence wedge вЂ” it risks fighting mature policy engines on their turf rather than sharpening the one thing nobody else does. **Counter to the counter:** (B) blocks a runtime thread on a subprocess inside `spawn_blocking` вЂ” the "no blocking in async"/thread-starvation smell rust-conventions warns against вЂ” and hides the subprocess from the scheduler and the hot-path timeout; (A) is the correct seam. And the dilution is bounded: the exec-verifier machinery already exists, so the marginal cost is only the permit-axis dispatch + the sync/async seam вЂ” nothing more. The latency answer is the deferred decision fast-path cache (permit-engine В§10.2), not fail-open; SP3 states the ceiling, it does not hide it.

## Decision

1. **Ratify extending the permit axis to dispatch exec verifiers (sketch В§2/В§3.1).** Un-filter exec entries in `permit_config_for` (they parse today but drop at `mcp.rs:157`); extend `PermitVerifierSpec` to carry an exec kind (argv + display name + params); extend `permit::aggregate` to dispatch exec entries through the EXISTING `ExecVerifier` (`rezidnt-gate/src/lib.rs:867`), preserving ordered first-`Fail`в†’Deny short-circuit, any-`Inconclusive`в†’Escalate, and the verdictв†’decision map. No new verifier machinery, no new vocabulary.
2. **Ratify async permit dispatch вЂ” option A (sketch В§3).** Lift permit aggregation to the async layer of `decide_permit`: natives stay in `spawn_blocking`, exec runs via `ExecVerifier::run().await`, ordered short-circuit preserved across native+exec. **Reject (B)** (`block_on`) вЂ” thread-blocking smell that hides the subprocess from the scheduler and the hot-path timeout.
3. **Determinism/replay is BINDING (I6, sketch В§4).** The policy artifact is content-pinned to CAS and its ref rides the decision fact's `policy_ref`; the В§8 contract records exact stdin/stdout so `debrief` re-executes the same policy bytes and compares (DR-006 integrity-alarm path on divergence). Exec runs network-off + scrubbed env (doc В§12): **a policy that fetches external data at decision time is NON-CONFORMING in SP3** вЂ” bundled data must be a pinned input, not a live fetch.
4. **rezidnt does NOT bundle a policy engine (I7, sketch В§5; owner-ratified scope В§9.3).** The engine is a local argv the operator provides (the exec-verifier model). SP3's acceptance is proven with a tiny reference policy argv (the oracle's deterministic judge); no OPA/Cedar binary enters the build. A concrete OPA-`eval`/Cedar adapter is a demand-gated follow-on with its own DR, not SP3.
5. **Ratify latency posture: defer-with-stated-ceiling (sketch В§6).** SP3 ships the one-shot argv dispatch and states the per-call ceiling (OPA/Cedar cold eval 10sвЂ“100s ms, at or above the SP2 250 ms budget); the decision fast-path cache (permit-engine В§10.2) is deferred, not required in SP3. Long-lived policy sidecar (OPA server mode) is out of scope вЂ” it reintroduces a network hop and stateful dependency (I7/determinism tension).
6. **Skip `/intel` for now (owner-ratified В§9.4).** The В§8 contract is engine-agnostic; defer any OPA/Cedar read to whenever a concrete adapter is built (its own DR then).

Verdict mapping (sketch В§7): an exec `VerifierOutput` maps `passв†’allow / failв†’deny / inconclusiveв†’ask` verbatim through the existing path (I6, never coerced вЂ” nonzero-exit / malformed / timeout в†’ `inconclusive` в†’ escalate, never a synthesized allow). The decision fact carries the exec verifier's `policy_ref` + `evidence_ref` so `gate_explain` surfaces the deciding external policy, not a hardcoded native.

This is scope + posture + the sync/async seam, not implementation. The build is a follow-on oracle-first slice (sequence: this DR в†’ `/oracle` в†’ implementer в†’ `/vet` в†’ `/debrief`).

## Consequences

- **В§16 roadmap delta:** SP3 is committed; acceptance per sketch В§8 вЂ” headline: a `[gates.permit]` block with an exec policy that DENIES a forced-breach request yields `deny` (the external policy decided), an allowing policy yields `allow`, proven with the reference policy argv.
- **В§8вЂ“В§9 delta:** the permit aggregate gains an exec dispatch path and `decide_permit` gains an async lift (natives stay in `spawn_blocking`). **No wire change and no ontology change** вЂ” SP3 reuses the existing gate/permit facts and the existing В§8 exec contract.
- **Risk-register (В§18) deltas:** *Hot-path latency* (permit-engine В§10.2) вЂ” the exec per-call ceiling (10sвЂ“100s ms, at/above the 250 ms budget) is **stated, not hidden**; the mitigation remains the deferred decision fast-path cache, not fail-open. *Determinism drift* вЂ” a policy reaching the network or reading unpinned data breaks replay; the sealed-env contract forbids it and a non-sealed policy is declared non-conforming. *Strategic dilution* (permit-engine В§10.3, restated) вЂ” SP3 is table-stakes DSL parity, bounded because the exec machinery already exists; the marginal cost is only the permit-axis dispatch + the sync/async seam.
- **Invariants touched vs untouched:** no invariant text is rewritten. **I6 determinism/replay is BINDING and upheld** вЂ” `policy_ref` pins the policy, `debrief` re-executes the same bytes, `inconclusive` never coerced to allow. **I7 upheld** вЂ” no bundled policy engine; the reference judge is a local argv. **I4** вЂ” exec dispatch is the existing substrate verifier capability. **I2** вЂ” bulky `VerifierInput`/evidence pins to CAS, never inline. **I1/I3** unchanged вЂ” the escalate/`ask` surface is a client; the decision fact remains the PDP path's output.

- **No test or acceptance criterion is weakened by this record.** In plain words: dispatching exec entries *enforces* policies that `permit_config_for` currently drops silently (a tightening вЂ” a declared exec permit gate now actually runs instead of being skipped); the never-coerce path keeps timeout/malformed/nonzero-exit at `ask`, never a synthesized allow; determinism pinning + replay is added, not relaxed. The only stated relaxation is the latency ceiling (В§6), which is disclosed, not a weakened criterion вЂ” the cache that would erase it is deferred with its ceiling on the record.
