# Decision Record DR-024 — Running-risk cap (C6): the deterministic risk scorer, pre-action placement, the granted-only fold, and the contract-free producer seam

**Date:** 2026-07-20 · **Status:** ACCEPTED (owner-delegated autonomy, 2026-07-20) · **Amends:** §8/§9 (permit engine — adds a `RiskCap` native beside `SpendCap` and makes the running-risk-score enforcement path LIVE) and the reducer **fold arm** for `risk_delta` (`rezidnt-state/src/lib.rs:750-752` — narrows it from all-outcomes to GRANTED-only). No invariant text is rewritten; this record SERVES I6, it does not amend it. **Cites:** intel memo [`intel/001-omnigent-permission-governance.md`](../../intel/001-omnigent-permission-governance.md) (C6 = a fenced own-DR permit capability; DR-002 rule 3, memo-motivated). **Builds on:** [DR-021](DR-021-live-spend-cap-c1.md) (C6 is the risk analogue of C1 and is judged against how C1 resolved the same shape — the post-action seam, the I3 honesty crux, the producer-seam §Context-C), [DR-008](DR-008-permit-engine-pivot.md) (PDP/PEP pivot, soft→ASK/hard→DENY verdict shape, I6 never-coerce), [DR-011](DR-011-permit-pdp-config-seam.md) (the PDP config/param-injection seam), SP5 accumulators/reducers (`PermitAccumulators.risk_score`, the `risk_delta`→`risk_score` fold).

## Context

`PermitAccumulators.risk_score` already exists (`rezidnt-state/src/lib.rs:163-166`) and `risk_delta` already folds into it in `apply_permit_decision` (`:750-752`), UNCONDITIONALLY for granted/denied/escalated alike. But nothing produces a `risk_delta`, nothing injects `cumulative_risk_score` into permit params, and no `RiskCap` verifier exists — the running-risk enforcement path is inert, exactly as C1's was before DR-021. DR-021 explicitly LEFT `risk_delta` on the permit path when it moved spend OFF (`:748-749`, DR-021 Consequences). C6 must decide whether that placement is right and complete the wiring. Five coupled questions, each judged against how DR-021 resolved the same shape.

**Q1 — the deterministic risk-scoring function (I6, the load-bearing question).** C1's cost was MEASURED/external (why DR-021 chose a post-action fact). C6's risk is COMPUTED here — the genuinely-new thing. It must be scored WITHOUT live inference (no LLM, no network, I6/I7), from inputs the PDP already pins on the request axis: `tool`, `paths`, `role` (`rezidnt-mcp/src/lib.rs:808-829`). Candidate shapes: (a) a rule/table scorer — per-tool base risk + path-sensitivity modifier + role modifier, summed; (b) a learned/model scorer (rejected on sight: live inference, non-replayable, a new dep — violates I6 and I7).

**Q2 — pre-action vs post-action (the C1 asymmetry).** Spend was scored AFTER the action (DR-021 A→3) because pre-action $ is unknowable. Risk is DIFFERENT: a rule/table scorer computes THIS action's risk deterministically from the request axis at decision time, before the action runs. So does C6 need the DR-021 post-action seam at all, or does `RiskCap` score inline from the request + read the cumulative from prior actions?

**Q3 — the denied-action honesty question (mirror DR-021 B2's crux).** Under I3 the folded `risk_score` is a durable, replayable claim the next decision reads. Today `risk_delta` folds for granted/denied/escalated alike (`:750-752`). A DENIED action never ran — folding its risk charges the run for danger it never incurred, the exact phantom-charge dishonesty DR-021 B2 refused for spend. Should the fold narrow to GRANTED-only?

**Q4 — the `RiskCap` verifier + config seam.** Mirror `SpendCap`: soft/hard caps ride the verifier's own `PermitVerifierSpec.params` (merged over the request axis, `permit.rs:410-425`); the PDP injects `cumulative_risk_score` from the folded accumulator (mirror the `cumulative_spend_usd` injection at `rezidnt-mcp:831-834`). Is this-action's risk computed INSIDE the verifier (from the request axis already in params) or injected?

**Q5 — the producer seam (mirror DR-021 §Context-C).** Q2 puts `risk_delta` on the `permit.granted` fact, but the score is COMPUTED inside `RiskCap` during aggregation, and nothing carries it out: `VerifierOutput` (`rezidnt-gate/src/lib.rs:95-100`) has no risk field and `PermitOutcome` (`permit.rs:342-377`) has no delta field, so the emit site cannot simply "produce a risk_delta." Something must carry the computed scalar from where it's decided to where it's stamped. This is exactly the seam DR-021 §Context-C treated as first-class; C6 cannot take DR-021's option-C-iii (no seam — spend went post-action), because risk RIDES the permit fact. So C6 must pick. (i) add `risk_delta` to `VerifierOutput` — a §8 exec-verifier STDOUT contract change churning every exec golden + replay. (ii) add a `risk_delta` field to `PermitOutcome` the aggregator populates from `RiskCap`'s output — natives-only, no §8 break (DR-021 option-C-ii analogue). (iii) NO plumbing: a SHARED deterministic `risk_score(request_axis, table)` pure fn called BOTH inside `RiskCap` (for its soft/hard verdict) AND at the emit site (to stamp `risk_delta` on the fact) — same fn, same content-pinned inputs, so the two calls yield the identical scalar by construction.

**Strongest counterargument (dissent, recorded verbatim per house style):** *"a hand-rolled deterministic risk score is security theater: a fixed per-tool table is trivially gameable and gives false confidence vs a real risk model, so shipping it invites over-trust in a number that doesn't mean much."* **Counter to the counter:** the alternative to a transparent, interrogable rule table is not a "real risk model" — that would be live inference, which I6 and I7 forbid outright (non-replayable, a new dep, coerces judgment into an opaque number nobody can audit). A rule table is the ONLY design compatible with the invariants, and its gameability is a FEATURE of its honesty: `gate_explain` names exactly which factors summed to the score, so an operator sees precisely what the cap does and does not cover and tunes the table against real behavior. The over-trust risk is answered not by a smarter number but by naming the model's limit plainly (Consequences) — the same discipline DR-021 used for lagging spend enforcement. The number is a coarse, honest, tunable heuristic, and it is presented as exactly that. **This is the owner's trade to make.**

## Decision

- **Q1 → option (a): a deterministic rule/table scorer.** `RiskCap` computes this action's risk from the pinned request axis: a per-tool base risk (config table) + a path-sensitivity modifier (touches a forbidden/sensitive path → add) + a role modifier, summed to a scalar. Pure Rust, no network, no inference. INTERROGABLE: the verdict's evidence NAMES each contributing factor and its weight (like `IntentLock` naming the off-task tool AND the declared intent, `lib.rs:807-812`), so `gate_explain` answers "why this risk". Option (b), a learned/inference scorer, is rejected: it breaks I6 (non-replayable) and I7 (new dep).

- **Q2 → PRE-ACTION, inline. C6 does NOT reuse the DR-021 post-action seam.** Risk is deterministically computable from the request before the action runs, so `RiskCap` scores this action inline and reads `cumulative_risk_score` (prior actions) from folded state. DR-021 went post-action ONLY because cost was unmeasurable pre-action — that constraint does not bind risk. `risk_delta` therefore LEGITIMATELY rides the permit fact: a granted action's deterministic risk is a real, non-guessed assessment, NOT the dishonest-if-denied estimate spend was. DR-021's `risk_delta`-stays-on-permit placement is confirmed as correct, not incidental.

- **Q3 → GRANTED-only fold (a reducer change).** Only actions that HAPPENED contribute running risk. `risk_delta` currently folds for granted/denied/escalated alike (`rezidnt-state/src/lib.rs:750-752`); this record narrows it to the `"granted"` arm. A DENIED or ESCALATED action never ran, so folding its risk is a phantom charge — the exact I3 dishonesty DR-021 B2 refused for spend. The permit fact honestly RECORDS the pre-action assessment; the accumulator honestly COUNTS only what ran.

- **Q4 → mirror `SpendCap`. This-action's risk is computed INSIDE the verifier.** Caps (`soft_cap_risk`/`hard_cap_risk`) + the scorer table ride `RiskCap`'s own `PermitVerifierSpec.params`, merged over the request axis (`permit.rs:417`). The PDP injects ONLY the folded `cumulative_risk_score` (mirror `rezidnt-mcp:831-834`); the current-action score is computed by the verifier from the `tool`/`paths`/`role` already on the request axis — NOT injected. Verdicts mirror `SpendCap` exactly: projected (`cumulative + this-action risk`) under soft → Pass; soft ≤ projected < hard → Inconclusive/escalate (NEVER coerced, I6, DR-008 §4); ≥ hard → Fail/deny; caps MISSING → cannot-run (garbage never coerces to a pass, I6). `RiskCap` registers in `builtin_natives()` beside `SpendCap` (`lib.rs:841-852`).

- **Q5 → option (iii): a single SHARED pure scorer, no producer-seam plumbing.** Extract `risk_score(request_axis, table) -> f64` as ONE deterministic pure fn. `RiskCap::verify` calls it for its soft/hard verdict; the emit site calls it to stamp `risk_delta` on the `permit.granted` fact via the EXISTING `DecisionDeltas` struct (`rezidnt-mcp:921-924` already threads `cost_ms` through it — `risk_delta` is a sibling scalar, no new carrier type). Both call sites hold the identical content-pinned inputs the merged params already expose (`deciding_params`, `permit.rs:363`, carries the request axis + caps/table config), so the two calls yield the SAME scalar by construction — the verdict and the folded delta CANNOT diverge. Rejected: **(i)** breaks the §8 exec STDOUT contract and churns every exec golden/replay for a NATIVE verifier that never touches that contract — an unjustified cost, and precisely the contract DR-021 option-C-iii fought to keep closed. **(ii)** is honest and natives-only, but threads a delta field through `PermitOutcome` and couples the folded scalar to the aggregator's plumbing when a shared pure fn makes the coupling unnecessary; it computes the score once but at the price of a new struct field and an aggregator write-back path. (iii) is preferred: its ONLY cost is one extra table lookup (a cheap pure computation on data already in hand), and in exchange it keeps `VerifierOutput`/`PermitOutcome` UNCHANGED, keeps the reducer folding a pre-stamped scalar (never running the scorer — no gate logic in the reducer, I3), and guarantees verdict/delta agreement structurally rather than by convention.

## Consequences

- **§8/§9 delta:** a new `RiskCap` native beside `SpendCap`; the PDP injects `cumulative_risk_score` from the folded accumulator (one insert mirroring `rezidnt-mcp:831-834`); `RiskCap`'s caps + scorer table ride its `PermitVerifierSpec.params` via the DR-011 config seam. **The producer seam (Q5) is contract-free:** a shared `risk_score` pure fn is called inside `RiskCap` for its verdict AND at the emit site to stamp `risk_delta` into the EXISTING `DecisionDeltas` (`rezidnt-mcp:921-924`) on the `permit.granted` fact. `VerifierOutput` (`lib.rs:95-100`) and `PermitOutcome` (`permit.rs:342-377`) are UNCHANGED — NO §8 exec-verifier STDOUT contract change, NO exec-golden/replay churn (the point DR-021 option-C-iii preserved for spend, preserved here for risk).
- **Reducer fold-source delta (§reducer):** `rezidnt-state/src/lib.rs:750-752` narrows from all-outcomes to the `"granted"` arm. This is a reducer change and a replay-visible behavior change on any existing log that folded denied/escalated risk — but no shipped log does, because nothing has ever produced a `risk_delta` (the path is inert), so no golden or replay churns.
- **Ontology / warden `/subject`:** `risk_delta` is an OPTIONAL scalar on the existing `permit.granted` payload — same grain and shape as the retired `spend_delta_usd?` and the live `cost_ms?`. Whether adding it needs a warden `/subject` is FLAGGED, not decided here: it is a new optional payload field on an existing subject, so it plausibly falls under the same gate the `permit.*` deltas did. The warden settles it; this record does not design the subject.
- **Risk-register (§18):** *over-trust risk (new)* — a coarse rule-table risk score may read as more authoritative than it is. Mitigated by naming it plainly as a tunable heuristic in product copy and by the interrogable factor breakdown; product copy must NOT claim a "risk model". *scope-gravity risk* — carried, mitigated by oracle-first slicing. C6 is CHEAPER than C1: no new subject grain, no post-action fact — it reuses the permit fact and the existing accumulator.
- **No shipped test or acceptance criterion is weakened.** The GRANTED-only fold narrowing does not soften any gate — it declines to charge phantom risk that was never charged in practice (the path was inert). New criteria arrive with the C6 slice via the oracle.

## Acceptance-criteria sketch (what `/oracle` encodes once the C6 build slice starts)

1. Config `[gates.permit]` carrying `soft_cap_risk`/`hard_cap_risk` + a scorer table causes the PDP to inject `cumulative_risk_score`; `RiskCap` runs (not cannot-run) and returns Pass when projected risk is under soft.
2. A request whose deterministic score (per-tool base + path + role) drives projected risk into `[soft, hard)` → Inconclusive → Escalate/ASK; ≥ hard → Fail → Deny.
3. The `RiskCap` evidence NAMES each contributing factor (per-tool base, path modifier, role modifier) — `gate_explain` surfaces the breakdown (I6 interrogability).
4. Caps missing from params → cannot-run (Inconclusive), never a synthesized pass (I6).
5. Same request axis → same score across two runs (determinism/replay); no network or inference is reachable from the scorer.
6. A GRANTED action folds its `risk_delta` into `cumulative_risk_score`; a DENIED or ESCALATED action folds ZERO (the Q3 honesty property, replayable from the log).
7. The `risk_delta` the reducer folds equals the score `RiskCap` used for its verdict on the SAME action — the shared `risk_score` pure fn guarantees they cannot diverge (assert equality of the stamped delta and the verdict's evidenced score, replayable from the log). `VerifierOutput` and `PermitOutcome` carry NO risk field (the seam is contract-free — asserted, not just the total).

## What this does NOT decide

- The exact numeric weights / the initial scorer table contents — a config/tuning concern, not an invariant, settled in the build slice.
- Whether `risk_delta` on `permit.granted` needs a warden `/subject` — flagged for the warden.
- Reconciling risk against C7 intent-lock or C8 layered precedence — those verifiers aggregate independently; no coupling decided here.
