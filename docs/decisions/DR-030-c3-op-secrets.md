# Decision Record DR-030 — the 1Password `op` SecretSource backend (`c3-op-secrets`): a headless `OpSecretSource` behind the ratified `SecretSource` I4 seam (DR-029) that resolves an `op://vault/item/field` reference to a secret value daemon-side by exec'ing the 1Password CLI (`op read`), authed by a vault-scoped service account — the second backend behind the seam DR-029 fenced, minting NO invariant change and NO new linked dependency

**Date:** 2026-07-21 · **Status:** ACCEPTED — ratified under the owner's standing high-autonomy grant (2026-07-20); the owner directed the op-CLI + service-account direction in a 2026-07-21 Q&A after reviewing the 1Password/Claude docs (see Context); the owner may revert. · **Amends:** §16 (carves a new slice, **`c3-op-secrets`**, after c3-egress-fold). **No invariant text is rewritten. No enforcement posture is changed. No new linked dependency. No new ontology is minted** (the `credential.dropped`/`credential.injected` subjects were minted in DR-029's paired warden `/subject`; this DR adds a resolution BACKEND, not a fact taxonomy). · **Builds on:** [DR-029](DR-029-c3-egress-fold.md) (the ratified `SecretSource` I4 trait `resolve(secret_ref) -> Result<Option<BrokeredSecret>, _>` at `crates/rezidnt-run/src/secret.rs`, the host-file MVP backend, and the `[egress]`/`[egress.secrets]` `host → secret_ref` project-spec surface — this is the `op` backend DR-029 §Decision 4 explicitly FENCED "as the next backend behind the same trait, exec'd, I7-clean; its own DR"), [DR-026](DR-026-c3bc-egress-credential-brokering.md) (the ratified credential-brokering POSTURE — secret injected upstream after TLS termination, agent NEVER holds it, logged by-ref `secret_ref`, redacted `BrokeredSecret`, the single audited `.expose()`, degrade CLOSED). · **Follows the precedent of:** [DR-025](DR-025-c3a-linux-sandbox.md)/[DR-001](DR-001-native-substrates.md) (the pasta/bwrap availability-probe pattern — a missing tool is a VERDICT, not a crash), and the git/pasta/bwrap "exec'd, not linked" I7 discipline. · **Cites:** no intel memo (a tool integration the owner chose, not a competitor read — no clean-room concern under I8/DR-002 rule 3; the product docs named below are 1Password/Anthropic first-party support pages, not a competitor codebase).

## Context

DR-029 shipped the `SecretSource` I4 seam and its host-file MVP backend, and explicitly fenced a 1Password `op`-CLI backend "as the next backend behind the same trait." This DR settles that backend. The host-file MVP is honest but crude: secret VALUES live in a host TOML the daemon reads; there is no rotation, no audit, no least-privilege scoping. A real deployment wants a managed store. The owner chose 1Password on 2026-07-21 after reviewing the docs.

**A product distinction had to be surfaced first, and is recorded here so it stays traceable.** "**1Password for Claude**" (support.1password.com/1password-claude; support.claude.com article 15936181) is a **human-in-the-loop connector** for Claude Desktop (macOS) + Claude in Chrome: it fills web-login credentials on the page "outside the agent's view" with per-task biometric approval. It has **NO programmatic API and NO `op://` syntax**. That is the right tool for the owner's INTERACTIVE Claude, and the WRONG tool for the headless rezidnt daemon (no human, no browser, no biometric). The daemon backend is 1Password's **developer tooling**: **`op read op://vault/item/field`** (headless, resolves to stdout), authed by a **service account** via **`OP_SERVICE_ACCOUNT_TOKEN`** (vault-scoped least-privilege, no desktop app, usage-auditable). The owner confirmed this direction after I surfaced the mismatch. The connector is explicitly OUT of scope (wrong layer).

The seam already fits: `resolve(secret_ref) -> Result<Option<BrokeredSecret>, _>` with `Ok(None)` as the DROP signal and `Err` as a genuine source failure (`secret.rs:42-46`). This DR adds a backend that exec's `op` and a scheme-dispatch so both backends coexist; it does not reshape the trait.

## Strongest counterargument (dissent, recorded verbatim per house style)

*"You're shelling out to a third-party binary on the hot path of every governed spawn, trusting its stdout is a secret and its exit code is truth. `op` needs the network to resolve, so now a 1Password outage or an expired service-account token silently degrades every mediated run to ConfinedClosed — you've coupled the product's core enforcement promise to an external SaaS's uptime and a token someone has to rotate. And you've added a second daemon-side secret (the service-account token) whose leak hands an attacker every vault the daemon can read. The host-file backend was ugly but it was self-contained and offline; this trades a local file for a network dependency and a bearer token, on the critical path, to save a rotation feature nobody has asked for yet."*

## Counter to the counter

The dissent lands three real hits — SaaS coupling on the hot path, a new bearer-token surface, and trusting an external binary's output — and the Decision answers each structurally. (1) **Resolution is daemon-side at FOLD time, not per-request on a spawn hot path.** `op read` runs in `fold_c3_policies` in the daemon's host namespace (which has internet), BEFORE the confined agent's netns is sealed — once per governed run at fold, not per egress connection. A 1Password outage degrades that run's affected mapping; it does not sit inline on live traffic. (2) **The degrade is LOUD and honest, and ConfinedClosed is the DR-026 fail-closed FLOOR, not a silent open.** An outage / expired token / `op` nonzero exit DROPS the mapping with a `credential.dropped` fact carrying a distinguishing reason (this is exactly DR-029's `Ok(None)`/`Err` unresolvable-drop discipline) — never a fake token, never a silent open. Fail-closed under a SaaS outage is the correct posture, not a regression. (3) **`op` is exec'd like git/pasta/bwrap — no supply-chain link, no SDK crate** (I7). Its exit code being a verdict is the same availability-probe contract DR-025 already trusts for pasta/bwrap: a missing/failing tool is a VERDICT, not a crash. (4) **The coupling is opt-in per `op://` ref, and the host-file backend REMAINS** for air-gapped / no-SaaS deployments — `op` is ONE backend behind the seam, dispatched by reference scheme, not the only one; a project that wants zero SaaS uses plain labels and never touches `op`. (5) **The service-account token is a real new surface and is treated as one** — vault-scoped least-privilege, env-only, never logged / never in a fact / never in the agent's `env_clear`'d + folded child. That mitigation is stated in the Decision, not assumed.

## Decision

Carve the slice **`c3-op-secrets`** (Linux/WSL, the daemon platform), settling five things:

1. **`OpSecretSource` backend behind the `SecretSource` trait.** For a `secret_ref` of the form `op://vault/item/field`, EXEC `op read op://vault/item/field` (`std::process`, captured stdout = the value) — **exec'd, not linked** (I7; the pasta/bwrap/git precedent; NO 1Password SDK crate). Captured stdout becomes `BrokeredSecret::new(secret_ref, value)` — value never logged, redacted at the type level, reachable only via the single audited `.expose()` (DR-026). **Trim the trailing newline `op` emits.** The `op` binary path is INJECTABLE (a ctor taking an op-binary path) so host tests drive a fake `op` script; no live 1Password in host CI.

2. **`op://` reference grammar + scheme-dispatch for `[egress.secrets]`.** An `[egress.secrets]` value that is an `op://vault/item/field` string IS the `secret_ref` `OpSecretSource` resolves. **Dispatch by reference scheme:** a `secret_ref` starting with `op://` routes to `OpSecretSource`; a plain label routes to the shipped `HostFileSecretSource`. Settle the shape as a **`CompositeSecretSource`** (scheme-dispatch: `op://` → op backend, else → host-file) so BOTH backends coexist and one project can MIX them. The `op://` path is a REFERENCE (vault/item/field NAMES), not a secret — so it rides the `credential.injected` fact as the `secret_ref` exactly as a plain label does (by-ref-never-value, DR-026 crit — stated explicitly).

3. **Auth via a vault-scoped service account (`OP_SERVICE_ACCOUNT_TOKEN`).** The daemon holds the token in its environment (vault-scoped least privilege); it is passed to `op` via env ONLY, treated as a daemon-side SECRET — **never logged, never in a fact, never in the agent's env** (the confined child is `env_clear`'d + folded; the service-account token must not leak in). Absent/empty token ⇒ `OpSecretSource` is UNAVAILABLE (honest) ⇒ an `op://` ref cannot resolve ⇒ DROP with a loud `credential.dropped` fact (DR-029's unresolvable-drop discipline), never a fake secret.

4. **Resolution is DAEMON-SIDE at FOLD time, before the sealed-netns spawn.** `op read` needs outbound network to 1Password; the daemon (host namespace, has internet) resolves at `fold_c3_policies` time, BEFORE the confined agent's netns is sealed — no conflict with the egress chokepoint (the daemon is the trusted broker; egress governance is for the confined AGENT, not the daemon). This sequencing is load-bearing and is stated explicitly.

5. **Honest degrade taxonomy — every failure is a DROP, never a fake token, never a silent open.** `op` binary absent (not on PATH) ⇒ **unavailable**; `OP_SERVICE_ACCOUNT_TOKEN` unset/invalid ⇒ **auth failure**; `op read` nonzero exit (item/field not found, no network, not authorized) ⇒ **resolution failure**. ALL ⇒ the mapping is DROPPED with a loud `credential.dropped` fact whose reason DISTINGUISHES them; the run proceeds mediated for whatever resolved (or ConfinedClosed if nothing did, the DR-026 fail-closed floor). This mirrors the pasta/bwrap availability-probe pattern (a missing tool is a VERDICT, not a crash).

## Consequences

- **§16 delta:** adds **`c3-op-secrets`** after **`c3-egress-fold`** — the slice adding the `OpSecretSource` backend + `CompositeSecretSource` scheme-dispatch behind the DR-029 seam. Gets its own `/oracle` → impl → `/vet` → `/debrief`; rides DR-026/029's ratified posture + seam, re-designs nothing.
- **§18 risk-register delta — service-account token (carried from DR-029's secret-surface risk):** a service-account token is a NEW daemon-side secret. Mitigations: vault-scoped least privilege; env-only (never logged / never in a fact / never in the agent's env); `op` exec'd in the daemon namespace only; resolution daemon-side pre-seal. The `op://` ref in a fact is a NAME, not a value.
- **§18 risk-register delta — SaaS-uptime coupling (NEW, opt-in):** a 1Password outage / expired token degrades affected `op://` mappings to a LOUD `credential.dropped` (fail-closed, DR-026 floor), never a silent open. Coupling is opt-in per `op://` ref; the host-file backend remains for air-gapped/no-SaaS deployments. Resolution is fold-time (once per run), not inline on live egress.
- **No new linked dependency** (`op` exec'd). No invariant/posture change. macOS/Windows: `op` is cross-platform, but this slice's live arm is Linux/WSL (the daemon platform); other-platform `op` behavior is not specially handled here.
- **No shipped test or acceptance criterion is weakened.** The DR-029 host-file backend + all its criteria stand verbatim; the DR-026 leak-discipline + fail-closed floor stand. This ADDS a backend + a scheme-dispatch — the host-file path is untouched and remains the default for a plain-label ref.
- **Deferred/flagged:** an MCP-based 1Password backend (heavier; not this slice); the **"1Password for Claude" connector is explicitly OUT** (wrong layer — interactive Claude Desktop/Chrome, not the headless daemon); `op` session-token (non-service-account) auth if ever needed for a human-run daemon.

## Acceptance criteria — `c3-op-secrets` (honest + falsifiable, host vs WSL)

1. **(host) scheme-dispatch.** An `op://…` `secret_ref` routes to `OpSecretSource`; a plain label routes to `HostFileSecretSource`; a `CompositeSecretSource` resolves BOTH in one project (a mixed `[egress.secrets]`). *(host-provable.)*
2. **(host, injected/fake `op`) `OpSecretSource` resolves.** With an INJECTED op-binary path (a fake `op` script — NOT a live 1Password), an `op://` ref resolves by exec'ing the op-shaped command and capturing stdout as a redacted `BrokeredSecret`; the trailing newline is trimmed; the value stays redacted under `Debug`/`Display`. *(host-provable.)*
3. **(host) honest degrade taxonomy.** `op` absent ⇒ unavailable; token unset ⇒ auth fail; `op` nonzero exit ⇒ resolution fail — EACH DROPS the mapping with a `credential.dropped` fact carrying a DISTINGUISHING reason, never a fake secret. `OP_SERVICE_ACCOUNT_TOKEN` appears in NO fact/log. *(host-provable, via the fake `op` script's controllable exit + env.)*
4. **(host) I2/DR-026 leak-discipline.** The resolved value is in NO fact (only the `op://` ref as `secret_ref`); `.expose()` remains the single call-site; the service-account token is never placed in the agent's env. *(host-provable.)*
5. **(WSL/live, OWNER-GATED) live op-injected mediated run.** With a real `op` + a real `OP_SERVICE_ACCOUNT_TOKEN` on the box, a governed run with `[egress.secrets] "github.com" = "op://…"` reaches the run-loop Mediated arm with the op-resolved token injected upstream; the agent never holds it. **This arm is GATED on `op` installed + a service-account token present** — honest SKIP when absent (like the pasta/bwrap gates). *(WSL netns box; needs the owner's `op` setup.)*

Criteria 1–4 are **host-provable** via an injectable op-binary path + a fake `op` script (no live 1Password in CI); criterion 5 needs the **WSL netns box AND the owner's live `op` + service-account token** — mirroring the DR-029 host/WSL split and the pasta/bwrap availability gates.

## Exit demo — `c3-op-secrets`

Two takes. **Host (CI-safe):** a project spec with `[egress.secrets] "github.com" = "op://Prod/github-token/credential"` + a plain-label ref in the same block; a fake `op` script on the injected path returns the token to stdout ⇒ the `op://` ref resolves to a redacted `BrokeredSecret` (newline trimmed), the plain label resolves via host-file, both fold into the injection-map — mixed backends in one run. Then each honest floor, loud: remove the fake `op` (unavailable), unset the token (auth fail), make the script exit nonzero (resolution fail) ⇒ each DROPS with a distinguishing `credential.dropped` reason, never a fake token; grep proves `OP_SERVICE_ACCOUNT_TOKEN` is in no fact/log. **WSL (owner-gated):** with a real `op` + service-account token, a governed `rezidnt open` run reaches github.com over the broker with an op-resolved token the agent never held (only `secret_ref = "op://…"` in the fact); skipped honestly when `op`/token absent.

## What this does NOT decide

- **The "1Password for Claude" connector** — explicitly OUT of scope; it is the human-in-the-loop Claude Desktop/Chrome credential-filler with no programmatic API, the WRONG layer for the headless daemon.
- **An MCP-based 1Password backend** — a heavier alternative behind the same seam; not this slice.
- **`op` session-token (non-service-account) auth** — only if a human-run daemon ever needs it; the service-account token is the sole auth mode here.
- **macOS / Windows `op` backends** — `op` is cross-platform, but the live arm here is Linux/WSL (the daemon platform); other-platform behavior is a later concern behind the same seam.
