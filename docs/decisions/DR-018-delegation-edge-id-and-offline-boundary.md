# Decision Record DR-018 — Delegation edge id derivation & the offline-attenuation boundary (SP4b)

**Date:** 2026-07-19 · **Status:** ACCEPTED (owner-ratified, TwofoldTech LLC, 2026-07-20) · **Amends:** [DR-017](DR-017-permit-macaroon-delegation-sp4b.md) §Decision 2 & §Decision 4 (how the delegation edge's parent/child `badge_id`s are derived); the `badge_id` derivation convention on `agent.spawned.badge_id` / `permit.delegated.{parent,child}_badge_id` (§12; `spec/ontology.md` ~191, ~410–411 — the **DR-005** `blake3(identifier)[..8]` convention). No invariant text is rewritten. **Cites:** [DR-017](DR-017-permit-macaroon-delegation-sp4b.md) §Decision 1/2/4; design sketch [`docs/design/permit-macaroon-delegation-sp4b.md`](../design/permit-macaroon-delegation-sp4b.md) §1, §4, §7; [DR-016](DR-016-permit-roles-sp4-slicing.md) §Decision 3 (role nominated as the SP4b delegation trigger); [DR-005](DR-005-badge-consolidation.md) (`badge_id = blake3(identifier)[..8]`).

## Context

Wiring SP4b surfaced a BINDING collision between two of DR-017's own ratifications. Both horns, precisely:

**The offline property (SP4b's raison d'être).** DR-017 §Decision 1/2 and design §1/§4 define delegation as **offline attenuation**: a holder mints a *narrowed* child badge with **no daemon round-trip and no root key**. The construction requires it: `sig₀ = keyed_hash(root_key, identifier)`, each caveat re-keys the running sig off the previous sig. A holder can append a caveat precisely because the running sig — not the root key — is the next hop's key. This is why SP4b exists over the DR-005 opaque token. `Macaroon::attenuate` (`crates/rezidnt-run/src/badge.rs`) implements it faithfully and **preserves the identifier**; changing the identifier would force `sig₀` back to the root key, destroying the property.

**The edge-naming requirement.** The `permit.delegated {parent_badge_id, child_badge_id, added_caveats, run}` subject (DR-017 §Decision 2) must name **two distinct ends** of the delegation edge; its oracle fixtures assert `parent_badge_id ≠ child_badge_id`. But `badge_id = blake3(identifier)[..8]` (DR-005). A faithful `attenuate` **preserves the identifier** → parent and child share an identifier → **identical `badge_id`** → the edge collapses to a self-loop and the fixtures fail.

**What shipped, and the honesty problem.** To get distinct ids, the first producer wiring did **not** call `attenuate`. It re-minted the child with a fresh identifier `<run>:role:<role>` using the daemon root key, copying the parent's caveats and appending the role caveat. That produces distinct ids and a monotone-narrower caveat set, but it is **not offline attenuation**: it needs the root key and the daemon, and the faithful `attenuate` method ships **unused**. Under **I3** the recorded fact must be honest about the mechanism; a root-key re-mint recorded as an attenuated delegation misrepresents what happened. The auditor's `/debrief` (2026-07-20) confirmed the emitted fact is honest *on the monotonicity axis* (the child capability is genuinely ⊆ the parent's) but declined `pass`: the property SP4b exists to prove is not exercised by any producer path.

## Decision (owner-ratified, both horns)

**(a) Edge / `badge_id` derivation — A2 RATIFIED: derive `badge_id` from the running sig, not the identifier.** `Macaroon::badge_id()` changes from `blake3(identifier)[..8]` to a derivation over the **running MAC sig** — concretely `hex(blake3(sig)[..8])` — which changes per appended caveat. Parent and child therefore differ by their caveat chains, **not** their identifiers, so a *true* same-identifier `attenuate` yields distinct parent/child ids AND preserves the offline property. The producer path replaces the root-key re-mint with `base_badge.attenuate(role_caveat)`. Rationale: only A2 keeps SP4b's raison d'être (offline delegation) intact — A1 buys distinct ids by making the child a root-key re-mint, shipping a "delegation" that isn't one.

- **`badge_id` shape is unchanged** — an 8-byte hex prefix, loggable, **never the token** (I2/§12). Only the **pre-image** changes (the running sig, hashed, rather than the identifier). We derive from `blake3(sig)` rather than raw `sig` bytes so no bytes of the MAC itself land on the fabric.
- **Not a payload `v+1`.** `agent.spawned.badge_id` and `permit.delegated.{parent,child}_badge_id` remain `string`, `v = 1`; `badge_id` is an opaque loggable identifier with no consumer that parses its pre-image, so re-deriving it is a definition change to `Macaroon::badge_id()`, not a schema break. The warden sanctions the ontology wording change (~191, ~410–411) in a `/subject` pass.

**(b) The offline-attenuation process boundary — DEFERRED to a named later SP4 sub-slice.** A2 makes attenuation **real at the daemon boundary** (the daemon calls `attenuate`, not `mint`, for the delegation edge). It does **not** yet move the attenuation into a separate *lead-agent process* — design §1/§7's "a holder attenuates offline, no daemon round-trip" story. In the current architecture the single daemon holds the base badge and attenuates on the sub-spawn path; the trigger fires on the SP4a `role` field (DR-016 §Decision 3). The **true holder-offline boundary** — an agent calling `spawn_agent` bearing its own badge, the daemon attenuating on the holder's behalf at the door, or a genuine agent-side `attenuate` — is named here as a **later SP4 sub-slice** (roadmap §16). SP4b's shipped scope is fixed to: **a real `attenuate` call at the daemon boundary + the `permit.delegated` audit fact.** The `permit.delegated` semantics and design §1/§7 are honest as written once the producer calls `attenuate` (a caveat *is* appended and the sig *is* re-keyed offline-of-the-root-key); the only deferred piece is *who* holds the badge, and that is stated so the acceptance bar does not overclaim holder-offline delegation.

## Consequences

- **Roadmap (§16) delta:** the true holder-offline attenuation boundary is a named **later SP4 sub-slice** (mechanism per (b)); SP4b's shipped scope is a real daemon-boundary `attenuate` + the audit fact.
- **Follow-on (warden `/subject`):** re-word the `badge_id` derivation in `spec/ontology.md` (~191 general note; ~410–411 the `permit.delegated` id fields) from `blake3(identifier)[..8]` to the sig-derived form, and correct the `permit.delegated` semantics (line ~156) so it describes a real `attenuate` (caveat appended + sig re-keyed) at the daemon boundary, with holder-offline named as deferred. Additive wording; every subject stays `v = 1`; no `SUBJECTS_V0` / drift-hash change.
- **Oracle + implementer (SP4b, oracle-first):** the SP4b `badge_id` assertions change to the sig-derived form; a test pins that a *true* `attenuate` under a **shared identifier** yields distinct parent/child `badge_id`s and a child capability ⊆ parent (the property, now real). The producer (`bins/rezidentd/src/runs.rs`) replaces `Macaroon::mint(root_key, "<run>:role:<role>", …)` with `base_badge.attenuate(role_caveat)`.
- **Two honesty notes from the `/debrief` (documented, not blocking):** (1) the child's `Role` caveat is **inert at the §12 door** — `check_badge` builds no `ctx.role`, so role narrowing is enforced by the SP4a permit PDP, not the badge door; `gate_explain`/docs must not read `permit.delegated` as "the door refuses a wrong-role child." (2) An **absent-`now`** `check_badge` call injects a wall-clock read not recorded on any fact, so a `debrief` replay of that call cannot reproduce the exact expiry evaluation from log inputs alone — acceptable as an enforcement-edge read, but stated so replay is not overclaimed.
- **Invariants:** **I6** — monotone narrowing holds (the child's caveat set is a strict superset of the parent's under `attenuate`); no widening. **I3** — the recorded fact now matches the mechanism (a real attenuation), at the daemon boundary. **I7** — **no new dependency** (sig derivation reuses vendored `blake3`). **I2** — `badge_id` stays an 8-byte loggable prefix, token never on the fabric.

*Amendments to this record require DR-019.*
