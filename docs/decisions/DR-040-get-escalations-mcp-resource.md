# Decision Record DR-040 — `get_escalations` read-only MCP resource (outstanding permit escalations over the surface)

**Date:** 2026-07-24
**Status:** ACCEPTED
**Amends:** §9 (adds one READ tool, `get_escalations`, to the MCP tool register, generated no-drift like `tail_events`/`board_view`); reaffirms — does not touch — I1 (zero pixels in core), I3 (log is truth), I5 (MCP-first), I6 (deterministic interrogable read). Adds one new view type (`EscalationRow`) alongside `BoardView` in `rezidnt-state`, following DR-039's precedent. Mints NO new invariant, subject, ontology fact, badge, PDP path, reducer arm, or daemon dependency. Does NOT touch `resolve_permit` (DR-033/035). Adds a §20 index row.

## Context

DR-038 ratified the `rezidnt-operator` GUI and bound it (Decision 4, I5) to talk over EXISTING surfaces only: any capability it exposes is a governed MCP tool/resource FIRST, then the button. DR-039 gave the GUI its fleet read (`board_view`) — which already carries a `permit_escalated` COUNT per run. But an operator resolving escalations from the UI needs the DETAIL behind that count: one row per outstanding escalation with its action/target/reason/policy_ref, so the GUI can list them and resolve each via the EXISTING `resolve_permit` write (DR-033/035). No MCP read surfaces that detail today. This record supplies it.

`get_escalations` is a READ, in the `tail_events`/`board_view` class (§12, amended by DR-005: reads are unbadged). It carries no authority, mutates nothing, and takes `GetEscalationsArgs { run: Option<String> }` — filter to one run, or all runs when absent. It is the governed read the GUI's escalation-resolution UI must ride (I5 — buttons follow tools); the write it feeds is the pre-existing DR-033 `resolve_permit`, untouched here.

**The finding that sizes the slice — PURE PROJECTION, no reducer/state change.** A recon confirmed the request_ids are ALREADY in folded state: `AgentRunState.permit_ledger` is a `BTreeMap<request_id, PermitLedgerEntry>` (`crates/rezidnt-state/src/lib.rs:393`), and an entry whose `decision == Some("escalated")` IS an outstanding escalation — the reducer arm `apply_permit_decision` (~line 1173) already keys each escalation by request_id with its action/target/reason/policy_ref. Surfacing them is therefore a pure read of existing derived state: NO new reducer arm, NO new field on `AgentRunState`, NO ontology/subject change. Data path (I3-clean): `replay(None)` → `rezidnt_state::fold -> Graph` → a NEW pure projection `escalations(&Graph, filter: Option<&str>) -> Vec<EscalationRow>` reading `permit_ledger.values().filter(|e| e.decision == Some("escalated"))` across `agent_runs` (honoring the optional run filter), deterministic in `BTreeMap` key order. One whole-log fold per call — a snapshot, same cost as `board_view`.

**The layering — a separate tool over extending `BoardView`, recorded.** `board_view` stays the fleet SUMMARY: it already carries the `permit_escalated` COUNT per run. `get_escalations` is the lazy DRILL-DOWN detail — a small payload, one row per outstanding escalation, operator-driven. Extending `BoardView`/`RunRow` with per-escalation detail would bloat EVERY board read with data nobody needs until they drill in. Cleaner and honestly separated: the board shows the count; this tool shows the detail. The new `EscalationRow { run, request_id, action, target?, reason?, policy_ref? }` (Serialize + Deserialize) lives with `BoardView` in `rezidnt-state`, sourced verbatim from `PermitLedgerEntry` (`crates/rezidnt-state/src/lib.rs:193`) + the run key + the request_id map key — re-interpreting nothing.

**Strongest counterargument (recorded, not just the outcome):** "you already have the count on `board_view`; add the escalation detail to `RunRow` and ship no second tool — one read, one schema." Rejected on two grounds. (a) It taxes every board read with drill-down detail the fleet glance does not want — the summary and the detail have different consumers and different cadence, and coupling them makes the common read heavier for the rare need. (b) The detail is operator-driven and lazy (fetched when a UI opens an escalation panel), which is exactly a distinct tool's shape, not a field on the summary. The honest cost of the separate tool — one more view type in `rezidnt-state` and one more no-drift schema — is small and correct: `rezidnt-state` is the materialized-state crate (DR-039 established the precedent), and an escalation row is materialized state.

## Invariant posture (enumerated, not waved past)

- **I1 (zero pixels in core) — PRESERVED.** `get_escalations` returns data, not pixels; no render dep enters any core crate. Same writer-freedom posture as DR-039 — the new view type is a pure data struct in `rezidnt-state`.
- **I3 (log is truth, state derived) — PRESERVED.** Pure `fold` + pure projection over the ALREADY-folded `permit_ledger`; NO new reducer arm, NO new derivation, NO field added. The escalation ledger already exists; this only reads it.
- **I5 (MCP-first) — HONORED, and this record is the mechanism.** The governed read exists in the governed repo BEFORE the DR-038 GUI's escalation-resolution UI lights up. The GUI mints no capability: the resolve is the pre-existing DR-033 tool; this is only the read that feeds it. Buttons follow tools.
- **I6 (deterministic, interrogable) — HONORED, and load-bearing here.** Same content-hashed log → same rows (deterministic `BTreeMap` key order). Escalations surface VERBATIM — never coerced to granted/denied — carrying `reason` + `policy_ref` for interrogability. This is the honest-inconclusive invariant: a READ must not resolve, and it does not.
- No badge, no write, no new PDP path, no new subject, no ontology change, no new dep.

## Decision

1. **Add `get_escalations`, a READ-ONLY MCP tool** returning the outstanding permit escalations as `Vec<EscalationRow>` (`run`, `request_id`, `action`, `target?`, `reason?`, `policy_ref?`). Unbadged (read-class, per §12 as amended by DR-005), in the `tail_events`/`board_view` class. Arg = `GetEscalationsArgs { run: Option<String> }` — one run when present, all runs when absent.
2. **Data path = `replay(None)` → `rezidnt_state::fold` → a NEW pure projection `escalations(&Graph, filter: Option<&str>)`** reading `permit_ledger.values().filter(|e| e.decision == Some("escalated"))` across `agent_runs`, deterministic in `BTreeMap` order. One whole-log fold per call (snapshot cost, same as `board_view`). NO new reducer arm, NO new `AgentRunState` field.
3. **Add one new view type `EscalationRow`** (Serialize + Deserialize) alongside `BoardView` in `rezidnt-state`, sourced verbatim from `PermitLedgerEntry` + the run key + the request_id map key. Re-interprets nothing. This does NOT extend `BoardView`/`RunRow` — the board stays the count-carrying summary; this tool is the drill-down detail.
4. **No-drift schema (§9 BINDING).** Add `#[derive(JsonSchema)] GetEscalationsArgs { run: Option<String> }` to `crates/rezidnt-types/src/mcp.rs` (pattern of `TailEventsArgs`/`BoardViewArgs`), advertise it in `tools_list()` via `schemars::schema_for!`, and extend the no-drift oracle at `crates/rezidnt-mcp/tests/jsonrpc_surface.rs` with a `get_escalations`/`GetEscalationsArgs` case so the served `inputSchema` MUST equal the generated one.
5. **This record does NOT touch `resolve_permit`.** Resolution rides the EXISTING DR-033 tool with its DR-035 broad-scope-requires-TTL guard. The coupling is named for the downstream UI (below), informational, not this DR's scope.

## Consequences

- **Roadmap (§9 / §19).** §9's tool register gains `get_escalations` (the drill-down read that DR-038's GUI escalation-resolution panel rides, feeding the DR-033 resolve write). No new phase; the change is one read tool + one view type.
- **Risk register.** ADDS one honest cost in plain words: `get_escalations` folds the whole log per call — a snapshot, acceptable and identical to `board_view`'s cost. If a hot path ever polls it, a `since`/incremental variant is the mitigation, filed as deferred (below), not built now. `rezidnt-state` grows one more "view" type — acceptable, it is the materialized-state crate (DR-039 precedent).
- **Downstream coupling (named, informational — not this DR's scope).** The GUI lists escalations from this read, then resolves each via the EXISTING `resolve_permit`, which carries DR-035's broad-scope-requires-TTL guard. That guard is unchanged; this record adds no resolution path.
- **Test/criterion honesty (plain words).** This record WEAKENS no test and lowers no bar. It TIGHTENS the surface: the no-drift oracle (`jsonrpc_surface.rs`) gains a `get_escalations`/`GetEscalationsArgs` case, so the served schema is machine-pinned to the generated one. No intel memo is cited by this record.
- **Deferred, named (not silently dropped).** (a) A `since`/incremental escalations read (snapshot-only now). (b) Pushing/streaming escalation updates (pull-only now, mirroring `tail`/`board_view`; no server-push). Each is its own DR if demand arrives.

Amendments to this record require DR-041.
