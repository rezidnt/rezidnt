# Decision Record DR-039 — `board_view` read-only MCP resource (fleet BoardView over the surface)

**Date:** 2026-07-24
**Status:** ACCEPTED
**Amends:** §9 (adds one READ tool, `board_view`, to the MCP tool register, generated no-drift like `tail_events`); reaffirms — does not touch — I1 (zero pixels in core), I3 (log is truth), I5 (MCP-first), I6 (deterministic interrogable read). The only structural change is an intra-repo hoist of a pure projection from `rezidnt-tui` down into `rezidnt-state`. Mints NO new invariant, subject, ontology fact, badge, PDP path, or daemon dependency. Adds a §20 index row.

## Context

DR-038 ratified the `rezidnt-operator` GUI and bound it (Decision 4, I5) to talk over EXISTING surfaces only: any capability it exposes must be a governed MCP tool/resource FIRST, then the button. Its live "runs / worktrees / subject-histogram" panels are exactly such a pending capability — the fleet state the read-only board renders has no MCP read path; a client can `tail_events` (raw envelopes) but cannot fetch the *derived* fleet view without re-folding the whole log itself. This record supplies that governed read.

`board_view` is a READ, in the `tail_events` class (§12, amended by DR-005: reads are unbadged). It carries no authority, mutates nothing, and takes an empty `BoardViewArgs {}` snapshot arg (full fold), matching `tail_events`' arg-struct pattern. Its data path is I3-clean and reuses machinery that already exists: `replay(None)` (reads bypass the mutating `McpSubstrate` trait and fold from `self.replay()` on the `Arc<Fabric>`, `crates/rezidnt-mcp/src/lib.rs:1645`) → `rezidnt_state::fold(events) -> Graph` (`crates/rezidnt-state/src/lib.rs:1239`) → `project(&Graph) -> BoardView` — a pure fold followed by a pure projection that carries derived state verbatim and re-interprets nothing. Cost is one whole-log fold per call, identical to the `rezidnt board` read: a snapshot, acceptable, noted below.

**The crux — the layering decision, recorded explicitly.** `project()` and the `BoardView`/`RunRow`/`WorktreeRow` types currently live in `rezidnt-tui` (`crates/rezidnt-tui/src/lib.rs:45-166`), a crate that by design depends ONLY on `rezidnt-state` + `rezidnt-types`. That writer-free dependency set IS the board's structural read-only proof (`crate_has_no_writer_dependency`, DR-031, promoted to a permanent I1 guard). `rezidnt-mcp` IS a writer (it links `rezidnt-fabric`). Therefore `rezidnt-mcp` must NOT depend on `rezidnt-tui` to reuse the projection — that would drag a writer into the board's dependency neighborhood and entangle the very proof DR-031 made permanent. **Resolution: hoist `project()` + `BoardView`/`RunRow`/`WorktreeRow` DOWN into `rezidnt-state`** (the pure calc / materialized-state crate). Both `rezidnt-mcp` and `rezidnt-tui` then reuse the one projection from `rezidnt-state` with zero new deps and zero cycles; `rezidnt-tui` keeps ONLY its ratatui `draw()`/render layer, and its `crate_has_no_writer_dependency` proof stays exactly as green as before (its dep set is unchanged).

**Strongest counterargument (recorded, not just the outcome):** the projection lives with the board today; the smaller diff is to have `rezidnt-mcp` reuse it in place. Two forms of that were weighed and rejected. (a) *`rezidnt-mcp` depends on `rezidnt-tui`* — breaks S5: it puts a writer crate into the board's dependency graph and makes the board's read-only proof depend on a writer's good behavior; DR-031 made that proof BINDING precisely so it cannot be softened by convenience. (b) *`rezidnt-mcp` re-implements `project()`* — mints a second copy of the fold-to-view logic, which drifts from the board's and violates single-source-of-truth (I3's "state is derived", one derivation). The hoist-to-`state` answer is strictly better than both: one projection, zero cycles, both consumers thin, and the read-only proof untouched. The honest cost of the hoist — `rezidnt-state` grows a "view" type — is small and correct: `rezidnt-state` is the materialized-state crate, and a fleet view is materialized state.

## Invariant posture (enumerated, not waved past)

- **I1 (zero pixels in core) — PRESERVED.** `board_view` returns data, not pixels; no render dep enters any core crate. The hoist keeps `rezidnt-tui`'s dependency set unchanged, so `crate_has_no_writer_dependency` — an I1 proof — is untouched.
- **I3 (log is truth, state derived) — PRESERVED.** Pure `fold` + pure `project`; the tool carries derived fields verbatim and re-derives/re-interprets nothing. Hoisting the projection into the derivation crate keeps ONE derivation of the fleet view.
- **I5 (MCP-first) — HONORED, and this record is the mechanism.** The governed read exists in the governed repo BEFORE the DR-038 GUI's runs/worktrees panels light up. Buttons follow tools.
- **I6 (deterministic, interrogable) — HONORED.** Same content-hashed log → same `BoardView` (deterministic key orders are pinned in `project`). It is a pure read; `permit.escalated` counts surface on their own and are never coerced (RunRow already preserves this).
- No badge, no write, no new PDP path, no new subject, no ontology change.

## Decision

1. **Add `board_view`, a READ-ONLY MCP tool** returning the `BoardView` projection (`events_folded`, `workspaces_open`/`closed`, `counts_by_subject`, `runs[]`, `worktrees[]`). Unbadged (read-class, per §12 as amended by DR-005), in the `tail_events` class. Arg = an empty `BoardViewArgs {}` (full-fold snapshot), mirroring `tail_events`' arg-struct pattern.
2. **Data path = `replay(None)` → `rezidnt_state::fold` → `project`.** Pure fold + pure projection; no re-interpretation; one whole-log fold per call (snapshot cost, same as the `rezidnt board` read).
3. **Hoist `project()` + `BoardView`/`RunRow`/`WorktreeRow` from `rezidnt-tui` into `rezidnt-state`.** Both `rezidnt-mcp` and `rezidnt-tui` reuse the one projection from `rezidnt-state`; `rezidnt-tui` retains only the ratatui `draw()` layer. No new deps, no cycles. `rezidnt-mcp` MUST NOT depend on `rezidnt-tui` (would entangle the DR-031 read-only proof), and MUST NOT re-implement the projection (drift). `BoardView` gains `Serialize` so the result serializes through `tool_ok()`.
4. **No-drift schema (§9 BINDING).** Add `#[derive(JsonSchema)] BoardViewArgs` to `crates/rezidnt-types/src/mcp.rs` (pattern of `TailEventsArgs`), advertise it in `tools_list()` via `schemars::schema_for!`, and extend the no-drift oracle at `crates/rezidnt-mcp/tests/jsonrpc_surface.rs:53` so the served `inputSchema` MUST equal the generated one.

## Consequences

- **Roadmap (§9 / §19).** §9's tool register gains `board_view` (the read that DR-038's GUI runs/worktrees panels ride). Unblocks DR-038's live-read panels without a new backchannel. No new phase; the change is one read tool plus an intra-repo hoist.
- **Risk register.** ADDS one honest cost in plain words: `board_view` folds the whole log per call — a snapshot, acceptable at fleet scale and identical to the existing `rezidnt board` read. If a hot path ever polls it, a `since`/incremental board is the mitigation, filed as deferred (below), not built now. `rezidnt-state` grows a "view" type — acceptable, it is the materialized-state crate.
- **Test/criterion honesty (plain words).** This record WEAKENS no test and lowers no bar. It does NOT relax the board's `crate_has_no_writer_dependency` proof — the hoist leaves `rezidnt-tui`'s dep set unchanged, and re-running `crates/rezidnt-tui/tests/read_only.rs` green is part of this slice's done. It TIGHTENS the surface: the no-drift oracle (`jsonrpc_surface.rs:53`) gains a `board_view`/`BoardViewArgs` case, so the served schema is machine-pinned to the generated one. No intel memo is cited by this record.
- **Deferred, named (not silently dropped).** (a) A `since`/incremental board (snapshot-only now — no incremental fold). (b) Pushing/streaming board updates (pull-only now, mirroring `tail`; no server-push). Each is its own DR if demand arrives.

Amendments to this record require DR-040.
