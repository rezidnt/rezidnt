# Decision Record DR-029 — the real egress fold (`c3-egress-fold`): sourcing a non-empty egress allowlist + brokered secrets from a real config surface (a project-spec `[egress]` block + a daemon-side `SecretSource` seam) so a live governed `rezidnt open` run reaches the run-loop MEDIATED arm end-to-end — activating the arm DR-028's honestly-minimal empty deny-all fold left dead

**Date:** 2026-07-21 · **Status:** ACCEPTED — ratified under the owner's standing high-autonomy grant (2026-07-20); the owner directed these decisions in a 2026-07-21 config-surface Q&A (see below); the owner may revert. · **Amends:** §16 (carves a new slice, **`c3-egress-fold`**, after c3-wire). **No invariant text is rewritten. No enforcement posture is changed. No new linked dependency. No ontology is minted in this DR** — the real `egress.*`/`credential.*` subjects are minted in a paired warden `/subject` session (flagged below), not here. · **Builds on:** [DR-026](DR-026-c3bc-egress-credential-brokering.md) (the ratified credential-brokering POSTURE — secrets injected upstream after TLS termination, agent never holds them, logged by-ref `secret_ref`, redacted `BrokeredSecret`, degrade CLOSED), [DR-028](DR-028-c3-wire.md) (the composition + the `fold_c3_policies`/`compose_spawn` seam + the run-loop Mediated arm this now feeds), [DR-024](DR-024-running-risk-cap-c6.md)/[DR-016](DR-016-permit-roles-sp4-slicing.md) (the C6 no-widening guard preserved end-to-end). · **Follows the precedent of:** [DR-020](DR-020-sp4c-wire-layered-permit-sourcing.md) (`REZIDNT_ADMIN_PERMIT` host-authority-file: a dev physically cannot self-grant; a set-but-missing/malformed file is an honest startup error, never silently empty). · **Cites:** no intel memo (this is a tool integration, not a competitor read — the 1Password direction is the owner's, no clean-room concern under I8/DR-002 rule 3).

## Context

DR-028 wired the C3 composition and the run-loop Mediated arm, but folded an **EMPTY egress allowlist** (deny-all) as honestly-minimal — the empty-allowlist ConfinedClosed downgrade at `runs.rs:1079-1081` means `compose_spawn`'s Mediated arm (`runs.rs:1093-1107`) is **dead in production**: only the substrate/enforce suite exercises live mediation, never a real agent run. DR-028 §"What this does NOT decide" #1 explicitly deferred "the real `[gates.permit]`/role fold field" to "a later slice." This is that slice. `fold_c3_policies` (`runs.rs:989`) hard-codes `EgressPolicy::from_folded_authority(Vec::new(), BTreeMap::new())` at `runs.rs:1020` — there is no config surface a non-empty allowlist or an injection-map can come from, so the mediated path C3 exists to deliver is unreachable by any shipped run.

The missing piece is twofold: **where the allowlist + host→secret_ref map are DECLARED**, and **where the secret VALUES come FROM**. The owner settled both in a 2026-07-21 config-surface Q&A: (1) the allowlist + injection-map keys live in a **new project-spec `[egress]` block** (declarations, repo-checked-in, alongside `gates` on `ProjectSpec` at `crates/rezidnt-run/src/spec.rs:14-23`); (2) the secret VALUES resolve daemon-side **behind a `SecretSource` I4 trait**, with a host secrets-file MVP backend shipped this slice (mirroring the `REZIDNT_ADMIN_PERMIT` authority boundary at `bins/rezidentd/src/main.rs:121-160`) and a 1Password `op`-CLI backend fenced as the next backend behind the same trait. `EgressPolicy` already has the right shape — private `allowlist: Vec<Destination>` + `injection_map: BTreeMap<String, BrokeredSecret>`, sole ctor `from_folded_authority` (`egress.rs:167-233`) — so this DR sources it, it does not reshape it.

## Strongest counterargument (dissent, recorded verbatim per house style)

*"You're bolting a secret store onto a daemon to light up a code path a test already covers — every credential system starts as 'just read a host file' and ends as the breach vector; and abstracting behind a `SecretSource` trait for a 1Password backend nobody has built yet is speculative generality that widens the attack surface before it earns it. The mediated arm is green in the enforce suite; you have proof the mechanism works. What you are adding is a new sensitive on-disk surface, a new env var, and a trait indirection whose only current implementation reads a TOML — all to move a test's green from the substrate level to the run-loop level. That is a lot of new attack surface for a re-parenting of coverage you already have."*

## Counter to the counter

The dissent lands two real hits — a secret surface is genuinely sensitive, and premature trait generality is a real smell — and the Decision answers both structurally, not by assertion. (1) **This is not test-only theater: the run-loop mediated path is the actual product promise C3 exists for.** DR-026 crit 4 (the agent reaches an allowlisted host with an injected token it never held) is currently proven only in the substrate suite; a shipped `rezidnt open` run cannot do it, because there is no allowlist source. "The mechanism works in a harness" is exactly the overclaim DR-028's own honesty guard was written to forbid — the product may not claim mediation until a *shipped run* enforces it. (2) **The MVP is the already-trusted in-tree pattern, not a new invention.** The host secrets-file backend is the `REZIDNT_ADMIN_PERMIT` authority boundary (`main.rs:121-160`) applied verbatim: env-pointed, living OUTSIDE any workspace spec, a dev physically cannot self-grant a secret, absent ⇒ empty, missing/malformed ⇒ honest startup error. Values NEVER touch the repo (the `[egress.secrets]` map holds only labels/`secret_ref`s), never touch a log (by-ref, redacted `BrokeredSecret`, the one `.expose()` on the upstream-write path), resolve daemon-side, inject upstream only. (3) **The seam is one small trait, and it earns itself now** — the owner declined a hardwired host-file pick precisely so the store is swappable; a single `fn resolve(secret_ref) -> Option<BrokeredSecret>` is the minimum honest abstraction over "the daemon holds the values, the agent never does" (DR-026), and it is the difference between a host-file that must be rewritten for 1Password and a backend swap. The `op` backend is FENCED (its own DR), not built — no speculative code ships, only the seam the MVP already needs.

## Decision

Carve the slice **`c3-egress-fold`** (Linux/WSL, the mediated fold), settling six things:

1. **`[egress]` project-spec block — the folded allowlist source.** A new `[egress]` block on `ProjectSpec` (alongside `gates`, `spec.rs:14-23`): `allowlist = ["github.com", …]` (hosts) + an `[egress.secrets]` table mapping `host → secret_ref` (a NAME/label, never a value — repo-safe). Parsed into the spec types; folded into `Vec<Destination>` + the `injection_map` keys. **Absent block ⇒ empty allowlist ⇒ deny-all** (DR-028's honest default preserved — absent NEVER means open; a malformed block is an honest error, never silently empty). The allowlist/map are PROJECT-DECLARED folded authority, NOT run-supplied — C6/DR-024 holds: the private-field `EgressPolicy::from_folded_authority` (`egress.rs:202`) stays the sole door; no `SpawnPlan`/request door is added.

2. **`SecretSource` I4 trait — daemon-side value resolution.** `fn resolve(&self, secret_ref: &str) -> Result<Option<BrokeredSecret>, _>`. The fold assembles the `injection_map` by, for each `(host → secret_ref)` in `[egress.secrets]`, resolving the VALUE via the `SecretSource` and building `BrokeredSecret::new(secret_ref, value)`. **A `secret_ref` the source cannot resolve ⇒ that mapping is DROPPED with a loud fact** — never a fake/empty secret, never an injection of the empty string; the run still proceeds mediated for hosts whose secrets resolved (or degrades per the resolved set). Selected like the other substrates (DR-001). The daemon is the sole holder of values (DR-026 posture — the agent never holds them).

3. **Host secrets-file MVP backend (shipped this slice).** `REZIDNT_EGRESS_SECRETS` → a host TOML (`secret_ref = "value"`), living OUTSIDE any workspace spec — the `REZIDNT_ADMIN_PERMIT` authority-boundary precedent (`main.rs:121-160`, DR-020). Absent env ⇒ empty source (no secrets resolvable ⇒ mediated-without-injection or deny per policy, honest). Set-but-missing/malformed file ⇒ an honest startup/fold error, never a silently-empty source that would drop the boundary. A dev cannot edit the host file — the authority boundary.

4. **1Password `op`-CLI backend — FENCED (next slice, its own DR).** Behind the same `SecretSource` trait; `op read op://vault/item/field` **exec'd, not linked** (I7-clean — the pasta/bwrap/git pattern; an MCP-based backend is the heavier alternative). Recorded here as the owner's stated direction so it stays traceable; **NOT built this slice.**

5. **Activate the run-loop Mediated arm end-to-end.** `fold_c3_policies` (`runs.rs:989`) folds the `[egress]` allowlist + the resolved injection-map instead of the `runs.rs:1020` empties; `compose_spawn`'s Mediated arm (`runs.rs:1093-1107`) becomes reachable when the fold yields a **non-empty allowlist + a live proxy**. The empty-allowlist ConfinedClosed downgrade (`runs.rs:1079-1081`) stays for the no-egress-config case (honest — deny-all, sealed netns, no fake mediation).

6. **Paired warden `/subject` (required, not designed here).** The live mediated run emits `egress.*`/`credential.*` facts that DR-025/026/028 left as PLACEHOLDER subjects. This slice PAIRS with a warden `/subject` session to mint the real subjects (`egress.mediated`/`egress.denied`/`egress.unavailable`/`credential.injected` — exact names the warden's call), each with a folding reducer (DR-006: no consumer-less subjects). Flagged as a required paired step; the warden designs the taxonomy, not this DR.

## Consequences

- **§16 delta:** adds **`c3-egress-fold`** after **`c3-wire`** — the slice that sources a non-empty `[egress]` allowlist + brokered secrets (project-spec block + `SecretSource` host-file MVP) so a shipped run reaches the run-loop Mediated arm. Gets its own `/oracle` → impl → `/vet` → `/debrief`; rides DR-026/028's ratified posture, re-designs nothing.
- **§18 risk-register delta — half-enforcement/overclaim (CARRIED, DISCHARGED for the mediated path):** now that run-loop mediation is REAL, DR-028's narrowed honesty guard is **discharged for the mediated path** — product copy may now claim per-platform (Linux/WSL) egress mediation + credential brokering for a *shipped* run with a configured `[egress]` allowlist + resolvable secret. Still degrade-CLOSED and loud on unavailability; the deny-all-when-absent and drop-on-unresolvable paths remain the honest floor.
- **§18 risk-register delta — NEW sensitive surface (secret store):** a secret store is a new attack surface. Mitigations: values NEVER in the repo (host-file/`op` only), never in a log (by-ref, redacted `BrokeredSecret`), resolved daemon-side, injected upstream only (the one `.expose()`); the `[egress.secrets]` map holds only `secret_ref` labels; the host file is env-pointed OUTSIDE any workspace spec (a dev cannot self-grant — the `REZIDNT_ADMIN_PERMIT` boundary).
- **No shipped test or acceptance criterion is weakened.** DR-026 crit 3/4 + CLOSED degrade, DR-028's three composed degrade facts + C6 no-widening, DR-025's loud-OPEN degrade all stand. DR-026 crit 4 gets *stronger*: it must now hold at the RUN-LOOP level (a shipped run), not only the substrate suite. The empty-allowlist deny-all default is preserved verbatim.
- **Deferred/flagged:** the 1Password `op` backend (next, its own DR); macOS/Windows egress backends (own DRs); the richer role-layer/`[gates.permit]`-precedence egress fold if ever demanded (this slice's `[egress]` block is project-level; admin/dev layering like DR-019/020 is a later concern); the `egress.*`/`credential.*` subjects are minted in a **paired warden `/subject`** (required, not designed here).

## Acceptance criteria — `c3-egress-fold` (honest + falsifiable, host vs WSL)

1. **(host) `[egress]` block parses.** The `[egress]` block (`allowlist` + `[egress.secrets]`) parses into the spec; an absent block ⇒ deny-all (empty allowlist); a malformed block is an honest error, never silently empty. *(host-provable.)*
2. **(host) `SecretSource` host-file backend resolves.** Resolves a `secret_ref → BrokeredSecret`; absent env ⇒ empty source; a set-but-missing/malformed file ⇒ honest error; an unresolvable `secret_ref` DROPS the mapping with a loud fact — never a fake/empty secret. *(host-provable.)*
3. **(host) fold is folded-only, no widening (C6 end-to-end).** The non-empty allowlist + injection-map reach `EgressPolicy` ONLY through `from_folded_authority`; the no-widening test still fails-FIRST if a `SpawnPlan`/request door is added; a run-supplied host/secret cannot widen it. *(host-provable.)*
4. **(WSL `#[cfg(unix)]`) live mediated run reaches the run-loop Mediated arm end-to-end.** A governed run with a non-empty `[egress]` allowlist + a resolvable secret: an allowlisted host is reached over the proxy WITH the injected token; an off-allowlist host is denied-and-logged; the token appears in NO log fact (only `secret_ref`); the agent never holds it — **DR-026 crit 4 now at the RUN-LOOP level, not just the substrate suite.** *(WSL netns box.)*
5. **(paired) facts ride real subjects.** The emitted facts ride the real minted `egress.*`/`credential.*` subjects (warden `/subject`), not placeholders. *(paired warden step.)*

Criteria 1, 2, 3 are **host-provable**; criterion 4 needs the **WSL netns box** (`#[cfg(unix)]` integration suite); criterion 5 is the paired warden `/subject` — mirroring the DR-027/028 host/WSL split discipline.

## Exit demo — `c3-egress-fold`

One take, on the WSL netns box. A real `rezidnt open` governed run against a project spec carrying `[egress] allowlist = ["github.com"]` + `[egress.secrets] "github.com" = "gh_token"`, with `REZIDNT_EGRESS_SECRETS` pointing at a host TOML resolving `gh_token`: the agent spawns confined AND **mediated** (pasta-outer, shared netns), reaches github.com over the broker with a token it never held (only `secret_ref = "gh_token"` in the fact), an off-allowlist host is denied-and-logged. Then the honest floor, each loud: drop the `[egress]` block ⇒ deny-all, confined + CLOSED; point `REZIDNT_EGRESS_SECRETS` at a `secret_ref` the file doesn't hold ⇒ that mapping dropped with a loud fact, never a fake token; malformed secrets file ⇒ honest startup error, never a silently-empty source.

## What this does NOT decide

- **macOS / Windows egress backends** — later, behind the same traits; Windows coupled to the deferred Platform phase.
- **The richer role-layer / `[gates.permit]`-precedence egress fold** — this slice's `[egress]` block is project-level; admin/dev layering (DR-019/020 style) is a later concern only if demanded.
- **The `egress.*`/`credential.*` warden subjects** — minted in a paired `/subject` session (required), not designed here.
