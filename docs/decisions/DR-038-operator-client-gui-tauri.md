# Decision Record DR-038 — Operator client GUI (Tauri), separate-repo

**Date:** 2026-07-23
**Status:** ACCEPTED
**Amends:** §19 (promotes the demand-gated "Tauri dashboard" open-decision to a ratified, separate-repo GUI operator client); reaffirms — does not touch — I1 (the daemon stays headless), I5 (MCP-first), I7 (one static daemon binary), I8 (clean-room applies to the client). Adds a §20 index row. Mints NO new invariant, subject, ontology fact, or daemon dependency.

## Context

The operator has, until now, driven rezidnt through the CLI (DR-036 onboarding), the read-only ratatui board (`rezidnt-tui`, S5), and the badged operator write path (DR-032 kill-run, DR-033 resolve-escalation over loopback MCP-HTTP). §19 has carried "First graphical client: ratatui board is S5; **Tauri dashboard is demand-gated**" as an open decision. The demand is now real: the owner wants an IDE-like operator surface to work on projects and inspect the daemon, first artifact a start screen — "Open a project" / "Look inside rezidnt."

**Form factor (owner-settled 2026-07-23): Tauri** — a Rust backend + web frontend. A plain static web page is the on-ramp and promotes to Tauri without a frontend rewrite (the frontend is the same web stack in both). A terminal UI was explicitly **rejected** for an IDE-scale surface: the read-only board proves ratatui's fit for a fleet glance, but an IDE workspace (panes, project tree, editors) is not a TUI-shaped problem. This does not disturb the board — the board stays the read-only fleet glance (DR-031); this is a distinct, richer surface.

**Where it lives (owner-settled): a separate repo/component, `rezidnt-operator`, sibling to the rezidnt repo — NOT inside the governed rezidnt crates.** This is the direct structural consequence of I1: the daemon renders nothing, and no rendering need may justify a daemon change. Keeping the GUI, its web toolchain, and its WebView2/Tauri stack entirely outside the workspace keeps I1 a fact about the tree, not a promise. It also continues the DR-031 board-read-only / operator-client split: the client is **write-capable** (operator actions), and the read-only board's structural writer-freedom (`crate_has_no_writer_dependency`, an I1 proof) must not be weakened by co-locating a writer near it.

**How it talks to the daemon:** over the surfaces that already exist — the MCP surface (I5) and the daemon's read/tail path for inspection, plus the badged operator write path (DR-032's loopback-HTTP lockfile + operator badge) for actions. **No new privileged backchannel is minted.** The client holds no authority the operator badge does not already grant; it is another badge-carrying MCP client, in a nicer skin.

**Strongest counterargument (recorded, not just the outcome):** "A GUI is Phase-3 scope gravity — the sequencing law is fabric → gates → terminal-last, and an IDE-scale front end is exactly the interactive-fidelity pull the plan warns against; ship product depth (benchmark harness, missing backends), not a dashboard." The force of this is real and is honored two ways, not by refusing the work: (1) the client is a **separate repo** that adds nothing to the daemon's critical path or dependency surface — building it cannot slow or complicate the governed core, so it is not the terminal-fidelity-before-fabric trap (the fabric and gates already exist and are being driven by real clients). (2) This record **gates only the live-daemon wiring**; design and the shell may proceed freely, but the demand-gate discipline of §19 is preserved by bounding what "authorized" means here (below). The counterargument correctly rejects putting pixels *in the core* — which is exactly why the answer is a separate repo, not a daemon change.

## Invariant posture (enumerated, not waved past)

- **I1 (zero pixels in core) — PRESERVED, and this is the load-bearing reason for the separate repo.** No pixels, no web toolchain, no WebView dependency enters the rezidnt crates. The client is a socket/MCP client like every other UI. The board's `crate_has_no_writer_dependency` proof is untouched — the GUI links no `rezidnt-tui` code and lives in a different repo.
- **I5 (MCP-first) — HONORED.** Every capability the GUI exposes is already an MCP tool/resource (or must become one *first*, in the governed repo, before it is a button). The GUI mints no capability of its own; if it needs a new operator action, that action is a governed MCP tool + DR, and the button follows. UI-second holds.
- **I7 (one static binary, no telemetry) — UNTOUCHED for the daemon.** The daemon remains one static binary (the DR-037 posture, plus its named literal-I7 follow-up). The GUI is a **separate artifact** in a separate repo; it is not "the binary" and does not amend I7's count. No telemetry / no phone-home is carried into the client (the invariant's spirit applied to the client too).
- **I8 (clean-room) — APPLIES to the client's sources.** `rezidnt-operator` is written clean-room: AGPL (herdr) sources are never read for it; permissive prior art only via `/intel`; nothing ported. No intel memo is cited by this record.

## Decision

1. **Ratify a first-party GUI operator client, `rezidnt-operator`, as a separate repo/component** sibling to the rezidnt repo — never inside the governed rezidnt crates (I1). This promotes §19's demand-gated "Tauri dashboard" line to a ratified surface.
2. **Form factor = Tauri** (Rust backend + web frontend); a plain static web page is the on-ramp that promotes to Tauri without a frontend rewrite. A terminal UI is recorded as rejected for this IDE-scale surface (the read-only board keeps the TUI role).
3. **First artifact = the start screen** — "Open a project" / "Look inside rezidnt" — the entry to an IDE-like workspace.
4. **Transport = existing surfaces only.** The MCP surface (I5) and the daemon read/tail path for inspection; the DR-032 loopback-HTTP lockfile + operator badge for write actions. **No new privileged backchannel; no authority beyond the operator badge.**
5. **This record GATES the live-daemon wiring, and only that.** Design, the separate-repo shell, and the static-web / Tauri prototype are FREE to proceed now. Wiring the client to a *live* daemon is what this DR authorizes and bounds: it must ride the badged MCP/read paths above and add no capability that is not first a governed MCP tool (per Decision 4 and I5).

## Consequences

- **Roadmap (§19).** Closes the "Tauri dashboard is demand-gated" open decision in favor of a ratified, separate-repo GUI operator client. No new phase inside the rezidnt repo; the governed roadmap is unchanged (the artifact is external). §19's operator-client register gains the `rezidnt-operator` GUI entry alongside the existing CLI operator subcommands (DR-032/033).
- **Risk register.** ADDS one honest risk in plain words: a first render/UI dependency stack (web + Tauri/WebView2) now exists in the project's orbit — but deliberately OUTSIDE the daemon, so it cannot reach the I1/I7 core; the mitigation IS the separate-repo boundary. Names the single authority boundary to keep honest: the operator badge is the only thing standing between the GUI and daemon mutation — the GUI must never grow a side channel that grants more than the badge (Decision 4 is the guard). Clean-room (I8) now also governs a web-stack repo — stated so it is not forgotten.
- **Test/criterion honesty (plain words).** This record WEAKENS no test and lowers no bar. It does NOT relax the board's `crate_has_no_writer_dependency` proof — the GUI is a different repo and links no board code. It adds no machine-checked guard of its own here (the client is external and un-vetted by the rezidnt gauntlet); that is stated plainly so no one mistakes a green rezidnt `/vet` for coverage of `rezidnt-operator`. Any GUI capability that needs a *new* operator action still owes that action a governed MCP tool + its own DR before the button exists (I5).
- **Deferred, named (not silently dropped).** (a) The `rezidnt-operator` repo's own CI / clean-room gauntlet — its own concern, outside this record. (b) Any *new* operator MCP tool the GUI turns out to need — each is its own governed DR + `/subject`, not folded in here. (c) The Tauri-vs-static-web packaging cutover — a prototype detail, not gated by this record.

Amendments to this record require DR-039.
